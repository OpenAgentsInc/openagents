Date: 2026-05-22

Master tracker: OpenAgentsInc/openagents#4520

- `OpenAgentsInc/openagents@da197710e4ea043f8e2b26c51d72a551fb7b6fcb`

- `OpenAgentsInc/treasury@a46d6d08fee19f91446b946c0fc45fc56af5a2e3`

## Decision

Pylon v0.2 is not release-ready for broad user distribution today.

The built-in Pylon LDK wallet implementation is substantially present and the
local regtest accepted-work payout proof is green. The release gate still fails
because the public `/pylon` operator copy is stale, the Autopilot4 deploy path
is blocked by non-interactive Google auth, the broad Pylon library suite has
nine red tests, and live Nexus treasury status is warning on stale wallet sync.

Do not claim "no known blocker remains" until the blockers in this report are
closed and verified from deployed state.

## Evidence

### OpenAgents Pylon

Current `main` includes the wallet runtime, backup/restore, channel readiness,
telemetry, accepted-work proof, and `external_payout_target` migration commits:

- `6163a5d42` - Add Pylon LDK wallet regtest harness
- `798619bc0` - Add Pylon accepted-work LDK payout proof
- `11f290863` - Add Pylon LDK wallet telemetry
- `78d02b99c` - Add Pylon wallet channel readiness surfaces
- `da197710e` - Migrate Pylon external payout target config

Important implementation facts:

- Default wallet runtime is `wallet_runtime_kind=ldk_node`.
- Wallet entropy is derived from the existing Pylon recovery phrase through a
  domain-separated HKDF path.
- Full Lightning recovery requires the encrypted wallet backup file and its
  passphrase.
- `external_payout_target` is now the explicit advanced migration override.
  Legacy `payout_destination` is retained only as a compatibility alias.
- Spark targets are rejected by active payout-target classification.
- Wallet status, telemetry, history, payment receipts, and harness summaries do
  not expose mnemonic words, raw node entropy, private keys, payment preimages,
  or bearer tokens.

### Autopilot4 `/pylon`

Repository `origin/main` contains the corrected built-in LDK wallet copy:

- `data/pylon-agent-instructions.md` references the built-in LDK wallet,
  encrypted wallet backups, `wallet channels --json`, and the external-target
  escape hatch.
- `src/pylon_page.rs` contains tests asserting the built-in LDK wallet,
  encrypted backup, and wallet-channel language.

Live deployment is stale. `https://autopilot4.openagents.com/pylon` still tells
agents to run:

```text
pylon config set payout_destination <lightning_target>
```

and still says the built-in LDK wallet is planned follow-up work. This is a
user-facing release blocker.

Deployment is blocked in this environment:

```text
gcloud builds list --project openagentsgemini --limit=1
gcloud run services describe autopilot4 --project openagentsgemini --region us-central1
```

Both fail with:

```text
Reauthentication failed. cannot prompt during non-interactive execution.
```

Open issue: OpenAgentsInc/autopilot4#142.

- `active_treasury_provider=ldk`
- `active_treasury_rail=ldk`
- `ldk_network=bitcoin`

## Required Blockers Before Release

1. Deploy Autopilot4 `main` and verify `https://autopilot4.openagents.com/pylon`
   no longer contains `pylon config set payout_destination <lightning_target>`,
   "external Lightning payout destinations are required", or "built-in wallet is
   planned follow-up work".
2. Resolve or explicitly quarantine the nine red `cargo test -p pylon --lib`
   failures with issue references and a release-owner signoff.
3. Refresh or reconcile live Nexus LDK treasury state so
   `/v1/treasury/status` no longer reports `stale_wallet_sync` before the
   release announcement.
4. Run the Autopilot4 build, test, deploy, and live smoke verification from
   authenticated deploy credentials.
5. After those changes, rerun:
   - `cargo test -p pylon --lib`
   - `cargo test -p pylon-tui --lib`
   - `scripts/pylon/ldk-accepted-work-payout-harness.sh`
   - Autopilot4 `/pylon` live copy smoke
   - Nexus `/v1/treasury/status` check

## Release Recommendation

Ship no public Pylon v0.2 distribution announcement yet.

The local LDK wallet path is strong enough to keep implementing against, and
the accepted-work proof now demonstrates the core receive, payout observation,
withdrawal, backup, restore, and channel-readiness path on regtest. The public
operator instructions and broad test gate still make this unsafe to present as
distribution-ready.
