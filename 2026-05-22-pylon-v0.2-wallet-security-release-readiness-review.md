## Decision

Pylon v0.2 is ready for controlled user distribution from the audited code and
deployed services.

The release blockers from the earlier audit are closed:

- the broad Pylon and Pylon TUI test gates pass;
- the regtest LDK accepted-work payout harness passes end to end;
- Autopilot4 `/pylon` is deployed with built-in LDK wallet instructions and no
  stale external payout destination requirement;
- Nexus is deployed on the LDK payer rail, reports connected wallet runtime and
  ready channel posture, and has no live failure, skip, or degraded treasury
  state;
- the final Nexus status endpoint is live-first rather than stale-cache-first.

The one nonzero backlog observed during final verification is not a stuck
release blocker. A live CS336 accepted-work run was still adding 25-sat payout
records while the audit ran. Over a full follow-up interval, the ledger advanced
from 400 to 401 payout records, confirmed payouts advanced from 399 to 400, and
the backlog stayed at one. That is steady drain under active workload, with
`pending_confirmation_count=0`, `tracked_payment_backlog_count=0`,
`payouts_failed_24h=0`, `payouts_skipped_24h=0`, and `degraded_states=null`.

### Treasury Repo Boundary

The `treasury` repo remains a planned closed-source custody-service surface. It
does not currently dispatch payouts or hold active spend authority. Active
production Lightning payment operations for v0.2 live in Nexus with LDK.

That boundary is acceptable for this release because user-facing Pylon custody
is built into the Pylon client and payer-side production dispatch is live in
Nexus. Treasury service extraction remains future work, not a blocker for
controlled Pylon v0.2 distribution.

## Telemetry Coverage

Required telemetry is present across the v0.2 path:

- Pylon CLI/TUI surfaces wallet status, registration mode, backup state,
  channel readiness, history, receive, send, restore, and telemetry snapshots.
- Pylon local telemetry and harness summaries omit wallet secrets.
- Nexus public stats expose provider, rail, wallet runtime status, sync lag,
  payout loop health, failed/skipped payout counts, degraded reason, and
  confirmed payout count.
- Nexus authenticated treasury status exposes LDK readiness, channel count,
  outbound capacity, wallet runtime, backlog, confirmation backlog, tracked
  payment backlog, ledger summary, active alerts, and degraded states.
- Autopilot4 `/pylon` gives operators the v0.2 wallet commands and recovery
  expectations that match implementation.

Known telemetry nuance:

- `backlog_total` can be nonzero while a live training run is actively creating
  accepted-work payout records. The release gate should consider backlog trend,
  confirmation progress, failed/skipped counts, pending confirmation count,
  tracked payment backlog, and degraded states together.

Do not frame this as a finished wallet product. Frame it as the v0.2 built-in
LDK wallet distribution candidate with:
