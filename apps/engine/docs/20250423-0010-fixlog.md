# Fix Implementation Log: PlanManager Service Not Found Error

## Problem Summary
After implementing the WebUI integration and AI service, the system encountered a critical error when processing GitHub issues: `Service not found: PlanManager`. This occurred despite successful server startup, API key loading, and SSE connection establishment.

## Diagnosis
The error indicated a fundamental issue with Effect.js dependency injection, specifically around Tag reference equality. Since Effect.js relies on JavaScript's reference equality for service Tags, the `PlanManager` Tag instance requested by `TaskExecutor` was not the same object instance provided by the Layer composition.

Multiple factors contributed to this error:
1. Missing `NodeContext.layer` in the AppLayer composition
2. Inconsistent export/import patterns for the PlanManager service
3. Potential stale compiled code affecting module resolution
4. Multiple imports creating separate Tag instances

## Implemented Fixes

### 1. Standardized Tag/Layer Exports 
In `src/github/PlanManager.ts`:
- Removed the `export const Default = PlanManagerLayer` line that created an alternative export
- Kept only the standard direct exports for `PlanManager` (Tag) and `PlanManagerLayer` (Layer)

```typescript
// BEFORE (at the end of the file)
export const Default = PlanManagerLayer

// AFTER
// Default export removed for consistency - use PlanManagerLayer explicitly
```

### 2. Standardized Imports in Server.ts
Ensured all imports were direct and consistent with usage patterns:

```typescript
// Consistent direct imports
import { PlanManager } from "./github/PlanManager.js";
import { PlanManagerLayer } from "./github/PlanManager.js";
```

### 3. Fixed NodeContext Layer in AppLayer
Added the essential NodeContext Layer to the Layer composition in Server.ts:

```typescript
return Layer.mergeAll(
  // Service layers
  GitHubClientLayer,
  PlanManagerLayer,  // Using direct import consistently
  MemoryManagerLayer,
  ContextManagerLayer,
  // Tools layer
  GitHubToolsDefault,
  // Task execution layer
  TaskExecutorLayer,
  // AI layer with HTTP client
  Layer.provide(AnthropicLayer, httpClientLayer)
).pipe(Layer.provide(NodeContext.layer)) // Added NodeContext layer
```

### 4. Performed Clean Rebuild
Removed all compiled files to ensure no stale module references:
```bash
rm -rf build
pnpm build-esm
```

## Verification
All TypeScript type checks pass with `pnpm check`. This solution aligns with Effect.js's design principles, which require strict reference equality for Tags used in dependency injection.

## Lessons Learned

1. **Tag Reference Equality**: Effect.js DI system relies on JavaScript object identity for Tags.

2. **Consistent Export/Import Patterns**: Using exactly one export/import pattern is critical.

3. **Layer Composition Order**: Certain layers like NodeContext.layer are foundational and must be provided at the right point in the layer composition.

4. **Clean Rebuilds**: For persistent DI issues, clean builds can resolve reference inconsistencies.

This fix ensures a solid foundation for the agent execution engine by correctly resolving all service dependencies.
