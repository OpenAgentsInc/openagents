# OpenAgents MDK And L402 Agent Checkout Audit

Date: 2026-06-02

## Scope

This audit covers how OpenAgents can add MoneyDevKit, abbreviated here as MDK,
for paid API access and agent checkout flows. The concrete target is:

```text
agent calls a protected OpenAgents endpoint
-> Worker returns HTTP 402 with an L402 challenge and Lightning invoice
-> agent pays the invoice
-> agent retries with L402 proof
-> Worker grants scoped access and records a receipt
```

The target runtime is the current Cloudflare-only OpenAgents stack in this repo:

```text
Foldkit web app
Cloudflare Worker API
D1 product/payment authority
Durable Objects for hot sync or realtime coordination
R2 for large artifacts
Queues/Workflows for asynchronous followup
OpenAuth and programmatic agent credentials for identity
```

This is not a replacement for the existing accepted-work payout architecture.
MDK is a checkout, L402, wallet-tooling, liquidity, and invoice helper. Pylon,
Nexus, and Treasury remain the authority for provider wallet identity,
accepted-work eligibility, company spend authority, and payout settlement.

## Short Recommendation

Use MDK first for **buyer-side agent payments into OpenAgents paid endpoints**.
Do not use MDK as accepted-work payout authority.

The first implementation should be:

```text
Cloudflare Worker
  -> authenticates a programmatic agent token
  -> applies a paywall policy to one endpoint
  -> returns bLIP-26-compatible L402 402 responses
  -> verifies L402 retry proof
  -> stores payment challenges, redemptions, entitlements, and receipts in D1

MDK
  -> creates/mints the Lightning invoice
  -> provides agent payer tooling through @moneydevkit/agent-wallet or pay402
  -> optionally provides hosted checkout/product dashboard behavior

D1
  -> source of truth for OpenAgents endpoint products, challenge state,
     entitlement state, one-shot redemption, and accounting refs
```

The practical v0 should adapt the MDK `withPayment`, `withDeferredSettlement`,
`pay402`, and token code patterns into an Effect-first Worker service rather
than importing Next.js-specific MDK wrappers into the Worker. The MDK Next.js
wrapper is useful source material, but this repo is a Foldkit/Vite app plus
Cloudflare Worker, not Next.js.

## 2026-06-05 Addendum: Site-Deployable MDK Checkout Primitive

The audit above is necessary but incomplete for Autopilot Sites. It treats MDK
mostly as an OpenAgents API paywall. The product direction needs a stronger
primitive:

```text
every Site generated by a human, agent, or Autopilot
-> can declare checkout products and paid actions
-> can deploy human checkout buttons/forms
-> can deploy agent-readable L402 paid APIs
-> can accept payment from MDK wallets or normal Lightning wallets
-> records payment evidence in OpenAgents product surface without storing MDK secrets in the Site
```

This should become a first-class Sites capability, not a one-off customer app
feature. A generated Site should be able to say "sell this product", "take a
deposit", "charge for this download", "gate this API action", or "let an agent
pay for this tool call" and receive working checkout surfaces from the
OpenAgents payment runtime.

The primitive has two related surfaces:

1. **Human checkout surface.** A generated Site renders a button, form,
   product card, tip jar, subscription card, or deposit/booking payment action.
   The browser calls an OpenAgents-hosted checkout-intent endpoint, which
   creates an MDK checkout and returns a redirect or hosted checkout URL. The
   Site receives clean success/cancel state by durable checkout records, not
   by leaking checkout state through public URLs.
2. **Agent checkout surface.** A generated Site exposes paid API actions in
   its manifest and OpenAPI. An agent calls the action, receives a standard
   `402`/`WWW-Authenticate: L402 ...` challenge, pays with an MDK wallet,
   `pay402`, or another Lightning wallet, retries with proof, and receives the
   protected resource or action result.

The generated Site must not own raw merchant credentials by default. In v0,
OpenAgents should be the hosted merchant-of-record/payment facilitator for
Sites and should store only redacted MDK refs, payment hashes, checkout IDs,
challenge IDs, receipt refs, and entitlement refs in D1. Later, customer-owned
MDK accounts can be supported through a reviewed secret-binding mode, but
`MDK_ACCESS_TOKEN`, `MDK_MNEMONIC`, webhook secrets, raw invoices, preimages,
wallet mnemonics, and payout credentials must not be emitted into source
archives, static JS, public manifests, screenshots, public proof, or model
visible generated files.

### Site Runtime Shape

Static R2 Sites:

```text
static Site button/form
-> POST /api/sites/:siteId/payments/mdk/checkouts
-> OpenAgents product surface creates checkout intent and MDK checkout
-> browser redirects to hosted checkout URL
-> OpenAgents product surface reconciles checkout and projects safe paid state
```

Worker/WFP Sites:

```text
Site Worker route
-> service binding or fetch to OpenAgents product surface Payment Worker
-> Site payment policy lookup by siteId/actionId
-> L402 challenge or checkout intent
-> protected Site handler
```

External or custom-domain Sites:

```text
custom domain component
-> OpenAgents hosted checkout intent URL
-> signed site/action metadata
-> clean return to canonical Site route
```

All three modes use the same product records, checkout intent records,
reconciliation events, receipts, and projection rules.

### Site Payment Manifest

Extend `.openagents/site.json` with an optional `payments` block. It should be
source metadata and generated-app intent, not a secret store.

```json
{
  "payments": {
    "enabled": true,
    "provider": "mdk",
    "merchantMode": "openagents_hosted",
    "checkoutBasePath": "/checkout",
    "products": [
      {
        "id": "consultation_deposit",
        "title": "Consultation deposit",
        "description": "Reserve an initial consultation.",
        "price": { "currency": "USD", "amountCents": 5000 },
        "kind": "one_time",
        "requiresCustomerData": ["email", "name"]
      }
    ],
    "paidActions": [
      {
        "id": "download_report",
        "method": "GET",
        "path": "/api/reports/download",
        "price": { "currency": "SAT", "amountSats": 100 },
        "settlementMode": "deferred",
        "agentReadable": true
      }
    ]
  }
}
```

This manifest block should drive validation, generated components, generated
route stubs, OpenAPI/payment manifest publication, and payment policy seeding.
It must reject secret-shaped values and unsafe return URLs the same way the
current site metadata helper rejects provider tokens and checkout state.

### D1 Product Model Additions

Add Site-specific product records beside the generic paid endpoint catalog:

```text
site_payment_products
site_payment_actions
site_checkout_intents
site_payment_events
site_payment_entitlements
site_payment_projection_records
```

These records link `siteId`, `siteVersionId`, `deploymentId`, `actionId`,

### Builder And Starter Requirements

Autopilot Sites, the self-serve builder, and the future SDK need reusable
payment primitives:

- generated checkout button/form/product-card components;
- generated checkout success/cancel pages that redirect to clean canonical
  URLs after state is reconciled;
- generated paid API route stubs for Worker/WFP Sites;
- generated OpenAPI and `/.well-known/openagents.json` payment capability
  entries;
- a small client helper such as `createMdkCheckoutIntent` that calls OpenAgents product surface,
  not MDK directly from browser code;
- a server helper or service binding for WFP Site Workers;
- sandbox/signet mode that exposes MDK sandbox signals and never asks agents
  to pay fake invoices;
- tests that prove no MDK secrets, raw invoices, preimages, wallet mnemonics,
  checkout query strings, or payment result state enter public source,
  manifests, URLs, projections, or logs.

### Updated Implementation Priority

The first MDK proof should still be a small paid endpoint, but the first
product-complete proof should be:

```text
generate a Site with one product and one paid agent action
-> deploy it as a static or WFP preview
-> click human checkout and create MDK checkout intent
-> call paid action as an agent and receive L402
-> pay/retry in sandbox or signet
-> reconcile payment evidence
-> unlock entitlement
-> show public-safe receipt/projection
```

This is the missing acceptance test for the broader Sites roadmap. MDK should
be a deployable component in every Site, not only an internal OpenAgents billing
feature.

## Existing OpenAgents Payment Position

The workspace docs already draw a clear boundary:

```text
Pylon owns contributor wallet identity and payout target.
Nexus owns accepted-work eligibility and public reconciliation.
Treasury owns spend authority and payout execution.
Autopilot/OpenAgents show operator workflow and receipts.
MDK helps with liquidity, checkout, payment-instruction parsing, and agent wallets.
```

That boundary should hold.

L402 paid endpoint access is a **customer pays OpenAgents** path. It is not the
same as **OpenAgents pays providers for accepted work**.

Therefore, the MDK/L402 layer should live beside the OpenAgents product API as
a paid-access gate, not inside Treasury's payout dispatch loop and not inside
Pylon's provider wallet authority.

## MDK L402 Protocol Findings

MDK's current docs describe L402 as a pay-per-call API layer:

```text
client requests protected endpoint
server returns HTTP 402 with invoice and signed token
client pays invoice and obtains preimage
client retries with Authorization: L402 <token>:<preimage>
server verifies token, resource, price, and preimage
handler runs
```

The official L402 spec describes the HTTP auth credential as:

```text
Authorization: L402 <macaroon(s)>:<preimage>
```

where the credential is carried over TLS and the preimage proves that the
Lightning invoice payment hash was paid. MDK's implementation uses the
`macaroon` wire field, but its token source file says the credential is an
HMAC-signed opaque token rather than a true macaroon. That is acceptable for a
pragmatic first OpenAgents integration if we treat it as an MDK-compatible
opaque credential and do not claim full macaroon caveat delegation until we
actually implement true macaroons.

Relevant MDK behavior to preserve:

- returns `WWW-Authenticate: L402 macaroon="...", invoice="..."`;
- accepts legacy `LSAT` auth scheme for compatibility;
- binds credential to `METHOD:/path`;
- freezes amount and currency into the credential;
- re-checks current endpoint price before granting access;
- verifies SHA256(preimage) equals the payment hash;
- returns 401 for malformed/invalid credential rather than minting another
  invoice;
- returns 403 for resource or amount mismatch;
- supports one-shot credential consumption;
- supports deferred settlement where the handler settles only after successful
  delivery.

The deferred settlement behavior is important for agent endpoints because an
agent may pay for a model call, sandbox action, crawl, or deployment action that
can fail after payment proof verification. Consuming the credential before the
handler succeeds creates paid failures with no retry. For expensive or
side-effectful OpenAgents endpoints, use deferred settlement.

## Proposed OpenAgents Flow

### Human checkout flow

This is useful for browser-facing paid upgrades, deposits, subscriptions, or
manual checkout pages.

```text
human clicks buy/deposit/upgrade
-> app calls Worker checkout route
-> Worker creates MDK checkout
-> browser redirects to checkout URL or renders a local checkout page later
-> success page verifies checkout/order state
-> Worker records payment evidence and grants entitlement
```

Use this for plan/subscription/deposit style flows, not for fast
machine-to-machine endpoint calls.

### Agent L402 flow

This is the core request.

```text
1. Agent registers programmatically.
2. Agent receives an `oa_agent_...` bearer token.
3. Agent calls protected endpoint:

   GET /api/paid/echo
   Authorization: Bearer oa_agent_...

4. Worker authenticates the agent.
5. Worker evaluates paywall policy for `GET:/api/paid/echo`.
6. If no valid L402 proof exists, Worker creates a challenge and returns:

   HTTP/1.1 402 Payment Required
   WWW-Authenticate: L402 macaroon="...", invoice="lnbc..."
   Content-Type: application/json

   {
     "error": { "code": "payment_required", "message": "Payment required" },
     "challengeId": "l402_challenge_...",
     "macaroon": "...",
     "invoice": "lnbc...",
     "paymentHash": "...",
     "amountSats": 100,
     "expiresAt": 1780444200
   }

7. Agent pays the invoice with:

   - `@moneydevkit/agent-wallet`;
   - MDK `pay402`;
   - its own Lightning wallet;
   - later, an OpenAgents-hosted budgeted agent wallet.

8. Agent retries:

   GET /api/paid/echo
   Authorization: Bearer oa_agent_...
   L402: <not used>
   Authorization: L402 <macaroon>:<preimage>

   The final header shape cannot use two `Authorization` headers reliably.
   Use one of the strategies below.
```

### Auth header collision

There is a practical HTTP issue: agent identity and L402 proof both want the
`Authorization` header.

Do not try to send two `Authorization` headers. Use one of these patterns:

```text
Preferred v0:
Authorization: Bearer oa_agent_...
X-OpenAgents-L402: <macaroon>:<preimage>

Strict L402 compatibility mode:
Authorization: L402 <macaroon>:<preimage>
X-OpenAgents-Agent-Token: oa_agent_...

Future structured mode:
Authorization: Bearer oa_agent_...
Payment: L402 <macaroon>:<preimage>
```

The endpoint should still return the standard `WWW-Authenticate` L402 challenge
so generic L402 clients know how to pay. For OpenAgents-owned agents, the
custom `X-OpenAgents-L402` proof header avoids breaking existing bearer auth.

If a third-party client sends only `Authorization: L402 ...`, then identity is
anonymous or must be recovered from a separate signed actor token. That is fine
for public paid endpoints, but private workspace endpoints should require an
agent or user identity first.

## D1 Data Model

Add payment tables in `workers/api/migrations/0003_mdk_l402_payments.sql` or a
nearby numbered migration. Keep names generic enough that MDK can be swapped

### Paid endpoint catalog

```sql
CREATE TABLE paid_endpoint_products (
  id TEXT PRIMARY KEY,
  stable_key TEXT NOT NULL UNIQUE,
  method TEXT NOT NULL,
  path_pattern TEXT NOT NULL,
  protected_ref TEXT NOT NULL,
  title TEXT NOT NULL,
  status TEXT NOT NULL CHECK (status IN ('active', 'disabled')),
  rail TEXT NOT NULL CHECK (rail IN ('mdk_lightning', 'manual_test')),
  currency TEXT NOT NULL CHECK (currency IN ('SAT', 'USD')),
  amount_sats INTEGER,
  amount_cents INTEGER,
  settlement_mode TEXT NOT NULL CHECK (settlement_mode IN ('immediate', 'deferred')),
  metadata_json TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE INDEX paid_endpoint_products_status_idx
  ON paid_endpoint_products(status);
```

Use exact path matching for v0. Add typed route matchers later. Do not add ad
hoc natural-language intent matching to choose paid endpoints.

### L402 challenge ledger

```sql
CREATE TABLE l402_challenges (
  id TEXT PRIMARY KEY,
  protected_ref TEXT NOT NULL,
  subject_user_id TEXT,
  subject_credential_id TEXT,
  subject_ref TEXT NOT NULL,
  method TEXT NOT NULL,
  path TEXT NOT NULL,
  resource TEXT NOT NULL,
  status TEXT NOT NULL CHECK (
    status IN ('issued', 'paid', 'redeemed', 'expired', 'revoked', 'failed')
  ),
  rail TEXT NOT NULL,
  currency TEXT NOT NULL,
  amount_sats INTEGER,
  amount_cents INTEGER,
  mdk_checkout_id TEXT,
  mdk_invoice_ref TEXT,
  invoice TEXT,
  payment_hash TEXT NOT NULL UNIQUE,
  token_hash TEXT NOT NULL UNIQUE,
  expires_at TEXT NOT NULL,
  paid_at TEXT,
  redeemed_at TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  metadata_json TEXT NOT NULL,
  FOREIGN KEY (subject_user_id) REFERENCES users(id),
  FOREIGN KEY (subject_credential_id) REFERENCES agent_credentials(id)
);

CREATE INDEX l402_challenges_subject_status_idx
  ON l402_challenges(subject_ref, status);

CREATE INDEX l402_challenges_protected_ref_idx
  ON l402_challenges(protected_ref, created_at);

CREATE INDEX l402_challenges_expiry_idx
  ON l402_challenges(status, expires_at);
```

Store `invoice` only if we accept the operational tradeoff. It is not a seed or
private key, but it is payment metadata. If privacy requirements tighten, store
only `mdk_invoice_ref`, `payment_hash`, amount, and a redacted invoice summary.

### L402 redemption ledger

```sql
CREATE TABLE l402_redemptions (
  id TEXT PRIMARY KEY,
  challenge_id TEXT NOT NULL,
  subject_user_id TEXT,
  subject_credential_id TEXT,
  protected_ref TEXT NOT NULL,
  status TEXT NOT NULL CHECK (status IN ('checked', 'settled', 'rejected')),
  request_id TEXT NOT NULL,
  response_status INTEGER,
  failure_code TEXT,
  created_at TEXT NOT NULL,
  settled_at TEXT,
  metadata_json TEXT NOT NULL,
  FOREIGN KEY (challenge_id) REFERENCES l402_challenges(id),
  FOREIGN KEY (subject_user_id) REFERENCES users(id),
  FOREIGN KEY (subject_credential_id) REFERENCES agent_credentials(id)
);

CREATE UNIQUE INDEX l402_redemptions_challenge_settled_idx
  ON l402_redemptions(challenge_id)
  WHERE status = 'settled';
```

This table gives us one-shot consumption and deferred settlement without
overloading the challenge row.

```sql
);

  ON payment_entitlements(subject_ref, entitlement_scope, status);

CREATE INDEX payment_entitlements_expiry_idx
  ON payment_entitlements(status, expires_at);
```

One-shot paid API calls may not need long-lived entitlements. Still create the
table because some paid surfaces will be subscriptions, deposits, paid
workspace scopes, or per-agent capability passes.

## Worker Runtime Notes

Cloudflare D1 is the right authoritative ledger for the first version because
it gives the Worker a serverless SQL database with SQLite semantics. Durable
Objects are useful for hot coordination and realtime state, but the payment
truth should live in D1 so replays, receipts, audits, and dashboards are not
tied to a warm object instance.

Do not blindly import the MDK Next.js package into the Worker. The reference
code uses Node-style APIs such as `process.env`, `Buffer`, `node:crypto`,
server package boundaries, and Next.js/Express route assumptions. Cloudflare
Workers can support some Node compatibility, but the cleaner implementation is
to port the narrow protocol logic from MDK core:

```text
MDK Next.js route    -> ignore as wrapper; port core route/action behavior
HMAC signing         -> Web Crypto or Worker-compatible helper
base64url/base64     -> Worker-safe encoder/decoder
env reads            -> Worker `env` bindings
checkout API calls   -> fetch client with Effect Schema decoding
Zod/oRPC contracts   -> Effect Schema models and stable OpenAgents product surface OpenAPI
D1 state             -> OpenAgents repositories, not MDK in-process globals
route wrappers       -> Worker middleware, not Next.js route exports
```

Concrete port map from the synced source:

- `packages/core/src/route.ts` should become explicit OpenAgents product surface Worker routes or a
  typed internal dispatcher. Do not expose MDK's browser-facing JSON
  `{ handler, ...payload }` multiplexer as the public product API.
- `packages/core/src/actions.ts` should inform checkout creation: amount vs
  product modes, metadata merging, customer-field normalization, checkout
  creation, confirm, and mint-invoice sequencing.
- `packages/core/src/client-actions.ts` should inform generated browser helper
  semantics, but OpenAgents product surface should call `/api/sites/:siteId/payments/...` endpoints
  instead of `/api/mdk` and should use OpenAgents product surface CSRF/session policy.
- `packages/api-contract/src/contracts/checkout.ts` and
  `packages/api-contract/src/schemas/checkout.ts` should be re-modeled with
  Effect Schema, preserving checkout status, amount/product/top-up shape,
  sandbox flag, customer data, L402 `check`/`redeem`, and programmatic payout
  result shapes where OpenAgents product surface exposes them.
- `packages/api-contract/src/validation/metadata-validation.ts` should be the
  upper bound for Site payment metadata: max size, key count, key length,
  alphanumeric/underscore/hyphen keys, no null/control characters. OpenAgents product surface must
  add its own secret-shaped key and value rejection.
- `packages/core/src/handlers/checkout.ts` should inform signed checkout URLs,
  query parsing, safe `checkoutPath` sanitization, create/get/confirm
  responses, and clean return redirects.
- `packages/core/src/mdk402/with-payment.ts` and
  `packages/core/src/mdk402/token.ts` should inform L402 challenge, credential,
  sandbox, price re-check, resource binding, preimage verification, one-shot
  redemption, and deferred settlement semantics.
- `packages/core/src/pay402.ts` should inform agent smoke tests: parse
  `WWW-Authenticate`, enforce `maxAmountSats`, use stable idempotency keys,
  wait for a preimage, and retry with `Authorization: L402 ...`.
- `packages/core/src/payment-state.ts` and webhook handling must not be copied
  as in-process global state. OpenAgents product surface needs D1 ledgers plus Queue/Workflow
  reconciliation so payment evidence survives Worker instance churn.
- `packages/core/src/handlers/pay_invoice.ts` is a preview-only shortcut and
  should remain sandbox/test gated. It is not acceptable as a production
  merchant-asserted "mark paid" route.

Use Durable Objects for:

- active paid workroom state;
- realtime payment/progress fanout;
- per-scope synchronization through OpenAgents Sync;
- short-lived coordination around one protected action if needed.

Use D1 for:

- challenge rows;
- redemption rows;
- entitlement rows;
- receipts;
- payment policy catalog;
- idempotency keys;
- agent budget ledgers.

Use Queues/Workflows for:

- expiration sweeps;
- webhook or checkout reconciliation;
- receipt export;
- periodic dashboard aggregation;
- retries around MDK status reads.

Do not run a long-lived Lightning node inside a normal Worker. If we need
`mdkd`, deploy it as an external service on SHC first, or later as a
Cloudflare Container if that becomes the chosen operational lane. The Worker
should call a narrow payment service, not own LDK channel state.

## MDK Deployment Choices

### Option A: Hosted MDK checkout/API first

Use MDK's hosted platform and SDK/API behavior for checkout/invoice creation.
Store only OpenAgents payment state in D1.

Pros:

- fastest path;
- least Lightning operations burden;
- aligns with MDK docs and dashboard;
- enough for the first paid endpoint.

Cons:

- dependency on MDK platform availability;
- Worker integration may need direct API/client adaptation because their
  packaged route helpers target Next.js or Express/Replit, not Cloudflare
  Worker;
- still need strong local D1 ledger because MDK is payment evidence, not
  product authority.

Recommendation: start here.

### Option B: `mdkd` on SHC

Run `mdkd` on SHC as an OpenAgents-managed payment service. The Cloudflare
Worker calls it over a private route, Tunnel, or authenticated public endpoint.

Pros:

- OpenAgents controls the daemon lifecycle;
- strong fit with the user's SHC direction;
- uses Rust LDK and local SQLite/VSS state;
- can expose invoices, sends, WebSocket notifications, and payment history.

Cons:

- introduces daemon ops, backup, monitoring, Basic Auth, and secret handling;
- more liability and custody complexity;
- not necessary to prove one L402 endpoint;
- must not be confused with Treasury payout authority.

Recommendation: use for a second phase if hosted MDK is insufficient or if we
want self-hosted L402 merchant infrastructure.

### Option C: Agent-local wallet only

Have each agent run `@moneydevkit/agent-wallet` and pay invoices returned by
OpenAgents.

Pros:

- excellent for agent demos;
- JSON CLI is simple for agents;
- no OpenAgents custody for payer funds;
- signet testing is straightforward.

Cons:

- not a server-side merchant checkout implementation;
- not enough for OpenAgents to mint invoices;
- each agent owns local wallet state and mnemonic hygiene.

Recommendation: use as payer tooling and test tooling, not as the server
merchant authority.

## Protocol Details To Implement

### Challenge creation

```text
Input:
  actor
  method
  path
  protectedRef
  amount/currency
  settlement mode
  expiry

Steps:
  create MDK checkout/invoice
  compute resource = METHOD:/path
  create signed L402 credential
  hash token before D1 storage
  insert l402_challenges row
  write receipt
  return HTTP 402 with WWW-Authenticate and JSON body
```

### Proof verification

```text
Input:
  actor
  request method/path
  L402 proof token and preimage

Steps:
  parse proof from X-OpenAgents-L402 or Authorization
  verify token signature
  verify token resource equals METHOD:/path
  verify current endpoint price equals frozen token price
  verify SHA256(preimage bytes) equals payment hash
  find challenge by token hash or payment hash
  confirm challenge is issued/paid and not expired/revoked/redeemed
  create checked redemption row
  call protected handler
  if immediate settlement, settle before handler
  if deferred settlement, settle only after handler succeeds
```

### Settlement

Immediate settlement:

```text
verify proof
mark challenge redeemed
insert settled redemption
grant one-shot entitlement if needed
run handler
```

Deferred settlement:

```text
verify proof
insert checked redemption
run handler
if handler succeeds:
  atomically mark redemption settled
  mark challenge redeemed
  grant entitlement if configured
if handler fails:
  leave challenge paid but unredeemed
  allow retry with same token/preimage until expiry
```

Use deferred settlement for:

- model calls;
- sandbox execution;
- deployment actions;
- expensive retrieval/crawl operations;
- any endpoint where OpenAgents promises a result, not merely admission.

Use immediate settlement for:

- cheap metadata;
- one-shot downloads where response creation is not expected to fail after
  verification;
- endpoints where "access attempt" is the purchased unit.

## Response Contract

Standard challenge response:

```http

Content-Type: application/json
Cache-Control: no-store

```

```json
{
  "error": {
    "code": "payment_required",
    "message": "Payment required"
  },
  "challengeId": "l402_challenge_123",

  "invoice": "lnbc...",
```

Sandbox/test response may add:


## Accounting And Authority Boundaries

Every paid endpoint event should be classified:

```text
payment challenge issued
payment proof observed
endpoint access delivered
entitlement granted
receipt recorded
```

Do not classify it as:

```text
accepted work
provider payout
Treasury settlement
Pylon wallet receipt
```

If the endpoint itself triggers work that may later produce provider payouts,
those payouts must flow through the existing accepted-work pipeline:

```text
paid endpoint request
-> OpenAgents receives payment
-> workroom/agent action runs
-> outcome reviewed/accepted
-> Nexus/Treasury payout eligibility
-> provider settlement receipt
```

The user/agent payment and provider payout may be causally related, but they
are different ledgers.

## Endpoint Candidates

Start with one intentionally low-risk endpoint:

```text
GET /api/paid/echo
price: 1 sat on signet or sandbox
settlement: immediate
actor: programmatic agent required
```

Then move to:

```text
POST /api/agents/runs/quote
price: 1 to 10 sats

```

Then:

```text

```

Do not put payment gates in front of auth callbacks, `/login`,
`/api/auth/session`, `/api/agents/register`, `/api/agents/me`, health checks,
or static assets.

## Client And Agent Tooling

### Generic agent

```ts
async function callPaidEndpoint(
  url: string,
  agentToken: string,
  payInvoice: (invoice: string) => Promise<string>,
) {
  const first = await fetch(url, {
    headers: { Authorization: `Bearer ${agentToken}` },
  })

  if (first.status !== 402) {
    return first
  }

  const challenge = (await first.json()) as {
    macaroon: string
    invoice: string
  }
  const preimage = await payInvoice(challenge.invoice)

  return fetch(url, {
    headers: {
      Authorization: `Bearer ${agentToken}`,
      'X-OpenAgents-L402': `${challenge.macaroon}:${preimage}`,
    },
  })
}
```

### MDK agent wallet payer

For local/signet testing:

```bash
npx @moneydevkit/agent-wallet@latest init --network signet
npx @moneydevkit/agent-wallet@latest balance
npx @moneydevkit/agent-wallet@latest send <bolt11-invoice>
```

The current agent wallet docs say commands output JSON and can pay BOLT11,
BOLT12, LNURL, or Lightning address destinations. The `send` output in local
docs currently returns a payment hash; a full L402 client needs the preimage.
If the CLI does not expose the preimage for a BOLT11 payment, use MDK `pay402`
or add an OpenAgents payer adapter that can obtain the preimage. L402 proof
requires the preimage, not merely a payment hash.

### MDK `pay402`

MDK's `pay402` helper already:

- fetches the endpoint;
- parses `WWW-Authenticate`;
- decodes invoice amount;
- enforces optional `maxAmountSats`;
- dispatches payment with idempotency;
- waits for preimage;
- retries with `Authorization: L402 <macaroon>:<preimage>`.

That is the best reference for OpenAgents agent-side payment behavior. For
OpenAgents-owned agents that also need bearer identity, wrap it to support the
`X-OpenAgents-L402` proof header.

- Effect Schema decoding for all payment payloads;
- L402 header parser;
- token creation/verification;
- preimage verification;
- canonical resource calculation;

MDK/signet smoke:

## Open Questions

1. Does hosted MDK expose a Worker-friendly direct API for the exact
   create-checkout-plus-mint-invoice flow, or should we initially wrap the
   package internals in a small server-side adapter?
2. Does `@moneydevkit/agent-wallet send` expose the payment preimage in the
   current release, or only the payment hash? L402 requires the preimage.
3. Should OpenAgents v0 credentials be MDK's HMAC opaque token for speed, or
   should we implement true macaroons immediately?
4. Should public paid endpoints allow anonymous L402-only access, or should all
   paid endpoint calls require a human/agent identity first?
5. Should invoice strings be stored in D1 for support/debug, or should D1 store
   only payment hashes and redacted invoice refs?
6. Should `mdkd` be deployed to SHC before or after the first hosted-MDK
   endpoint proof?

## Decision

Recommended path:

```text
1. Keep Pylon/Nexus/Treasury payout authority unchanged.
2. Add OpenAgents payment packages and D1 ledger tables.
3. Implement L402 middleware in the Cloudflare Worker.
4. Start with one paid agent endpoint.
5. Use hosted MDK for invoice creation first if possible.
6. Use @moneydevkit/agent-wallet and MDK pay402 as payer/test references.
7. Make the second proof a generated Site with one human checkout product and
   one agent-paid L402 action.
8. Deploy mdkd to SHC only when the first paid endpoint proves the protocol
   and we need self-hosted Lightning merchant operations.
```

This gives OpenAgents the machine-payable endpoint layer the user wants without
collapsing payment evidence, checkout state, provider payouts, and Treasury
authority into one unsafe abstraction.
