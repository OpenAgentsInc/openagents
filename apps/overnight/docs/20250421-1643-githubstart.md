# GitHub API Integration for Overnight Coding Agent

## Implementation Overview

This implementation creates a fully-functional GitHub API client with file fetching capabilities built using Effect's functional patterns. The code demonstrates proper error handling, dependency injection, and testability in the Effect ecosystem.

### Components Built

1. **GitHub API Client Interface**
   - Defined a clean interface for GitHub API operations
   - Implemented as an Effect Context Tag for proper dependency injection
   - Started with file fetching functionality

2. **Error Handling**
   - Created typed error hierarchy using Effect's `Data.TaggedError`
   - Implemented specific error types:
     - `GitHubApiError`: Base error class for all GitHub API errors
     - `FileNotFoundError`: For 404 errors when a file doesn't exist
     - `RateLimitExceededError`: For API rate limit errors including reset time

3. **Schema Definitions**
   - Used `@effect/schema` for type-safe validation
   - Defined schemas for both request payloads and responses
   - Added descriptive annotations for better documentation

4. **Mock Implementation**
   - Created a testable mock implementation of the GitHub API client
   - Built simulated responses for various scenarios (success, not found, rate limit)
   - Designed for testing without requiring actual API calls

5. **Comprehensive Tests**
   - Unit tests covering all functionality including happy paths and error cases
   - Tests for successful file fetching
   - Tests for each error condition (file not found, rate limit, server error)
   - Tests for schema validation failures

6. **Sample Program**
   - Demonstrated practical usage of the client in a real application
   - Showed proper error handling and response processing
   - Implemented base64 decoding for file contents

## Usage Guide

### Fetching a File from GitHub

```typescript
import * as Effect from "effect/Effect"
import * as Either from "effect/Either"
import { 
  createGitHubApiClient, 
  GitHubApiClient, 
  FileNotFoundError
} from "./github/GitHubApi.js"

// Create a program that uses the GitHub client
const program = Effect.gen(function*() {
  // Get the GitHub client from the context
  const githubClient = yield* GitHubApiClient
  
  // Fetch a file
  const fileResult = yield* Effect.either(
    githubClient.fetchFile({
      owner: "repoOwner",
      repo: "repoName",
      path: "path/to/file.md",
      ref: "main" // Optional: branch, tag, or commit SHA
    })
  )
  
  // Handle the result
  if (Either.isLeft(fileResult)) {
    const error = fileResult.left
    
    if (error instanceof FileNotFoundError) {
      yield* Effect.log(`File not found: ${error.message}`)
    } else {
      yield* Effect.log(`Error: ${error.message}`)
    }
  } else {
    const file = fileResult.right
    const content = Buffer.from(file.content, "base64").toString("utf-8")
    yield* Effect.log(`File fetched: ${file.name}`)
    yield* Effect.log(`Content: ${content}`)
  }
})

// Create a layer with GitHub client implementation
const MainLayer = createGitHubApiClient(
  "https://api.github.com", 
  "github_token_here" // Optional: add token for authenticated requests
)

// Run the program with the layer
Effect.runPromise(
  program.pipe(
    Effect.provide(MainLayer)
  )
)
```

## Technical Decisions

1. **Why Effect for GitHub API?**
   - Functional error handling with typed errors
   - Dependency injection through Context and Layers
   - Composable effects with proper cancellation and retry semantics
   - Schema-based validation for type safety

2. **Mock First Approach**
   - Started with a mock implementation to enable immediate testing
   - Designed for easy replacement with actual HTTP client
   - Allows testing without hitting rate limits

3. **Error Hierarchy**
   - Created specific error types to enable precise error handling
   - Used discriminated unions for type-safe error matching
   - Included relevant contextual information in errors

4. **Testing Strategy**
   - Focused on testing both success and failure paths
   - Used dependency injection to provide test implementations
   - Implemented comprehensive test coverage for all edge cases
