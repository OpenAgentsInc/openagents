Date: 2026-06-09

Status: audit for the new standalone `OpenAgentsInc/pylon` repo after Pylon
was extracted out of the `openagents` repo.

Update: after commit `4dba938`, Probe's runtime has been ported into this repo
as `packages/runtime` / `@openagentsinc/pylon-runtime`, with the public
`pylon` binary routing runtime namespaces such as `apple-fm`, `backend`,
`chat`, `auth`, and `openagents`.

`docs/2026-06-09-pylon-psionic-ml-connection-audit.md`. The v0.3 launch path
must keep GEPA-first benchmark rollout work separate from neural training, and
must block Qwen/training claims until Psionic connector, sidecar, artifact, and
receipt gates exist.

## Executive Verdict

The new standalone Pylon repo is the correct home for Pylon v0.3, and it now
contains the former Probe runtime surface needed for local agent/provider work.
It is still not launch-equivalent to the old public Pylon network surface.

The current standalone repo contains:

- package identity `@openagentsinc/pylon` at `0.3.0-rc1` until the launch
  gates pass;
- Bun, Effect, and OpenTUI dependencies;
- one OpenTUI app in `src/index.ts`;
- a fullscreen observational dashboard skeleton;
- a mocked hardware telemetry loop;
- an MDK `agent-wallet balance` poller;
- a stub presence heartbeat loop;
- an OpenCode diagnostic/composer path;
- a workspace runtime package, `@openagentsinc/pylon-runtime`, ported from
  Probe;
- Apple Foundation Models bridge client/readiness/streaming tool callback
  support;
- Gemini direct API and OpenAgents product surface-brokered Gemini materialization support;
- provider-neutral LLM request/message/tool/usage contracts;
- Blueprint signature lookup, tool menu planning, contribution gates, and
  Action Submission boundaries;
- GEPA/Terminal-Bench candidate execution, retained fixtures, closeout writer,
  token telemetry, runner identity, and OpenAgents product surface account/grant/provider-account
  contracts;
- retained Markdown/OpenTUI rendering helpers and streaming fixtures.

It does not yet contain the launch-critical public Pylon behavior from
`openagents` and the live network boundary:

- release/bootstrap/update semantics from `@openagentsinc/pylon`;
- Pylon home/config/identity persistence;
- NIP-98 account linking and refresh;
- public provider presence registration and heartbeat payloads;
- wallet readiness, payout-target admission, and send-readiness state;
- assignment polling, lease acceptance, progress, artifact/proof, closeout,
  stale lease handling, and receipt projection;
- `pylon-core` public-safe projection validation;
- live OpenAgents endpoint integration for the benchmark/GEPA worker loop;
- Qwen/training capability and receipt mapping, now explicitly postponed until
  after the GEPA-first v0.3 launch path;
- local ledger, earnings, payouts, settlement records, and wallet history;
- sandbox execution profile contracts and receipt gates;
- macOS/Linux install smoke coverage beyond the current repo skeleton.

The launch promise set therefore has to be reconfigured from "Probe plus the
old `openagents` Pylon launcher and deprecated Rust crates prove this" to "the
new standalone `OpenAgentsInc/pylon` repo owns the contributor node, contains
the local runtime, and must emit live public-safe network evidence." Until the
registration, heartbeat, wallet-readiness, assignment, closeout, and payment
gates are implemented against OpenAgents endpoints, public copy must keep
saying v0.3 is an extracted Pylon runtime in progress, not a broadly live
install-to-Bitcoin network.

## Boundary Change

### Old Assumption

The June 8 launch audit and related gates were written around:

- `@openagentsinc/pylon@0.2.5` as a bootstrap launcher inside
  `openagents/packages/pylon-bootstrap`;
- deprecated Rust implementation homes under `openagents/apps/deprecated/pylon`
  and `openagents/apps/deprecated/pylon-tui`;
- public contract helpers in `openagents/crates/pylon-core`;
- release and proof scripts in `openagents/scripts/pylon` and
  `openagents/scripts/release`.

### New Required Assumption

Pylon v0.3 must be treated as:

- standalone repo: `OpenAgentsInc/pylon`;
- runtime package: `@openagentsinc/pylon`;
- initial supported platforms: macOS and Linux only;
- implementation stack: Bun + Effect + OpenTUI;
- contributor-owned public supply connector;
- source of truth for local identity, telemetry, wallet readiness, assignment
  execution, artifact/proof closeout, and local public-safe receipts;
- not a private Cloud node, not Treasury, not buyer UX, not a hidden labor
  runtime, and not a browser wallet bridge.

## Required Ports Into `OpenAgentsInc/pylon`

### 1. Release, Bootstrap, And Install Surface

Port or replace from `openagents/packages/pylon-bootstrap`:

- macOS and Linux target resolution;
- GitHub release tag policy;
- install-root/cache layout;
- update polling while the dashboard is open;
- source-build fallback policy, if still desired;
- `--register-openagents`;
- `--setup-mdk-wallet`;
- public-safe `--pylon-ref`, display name, resource mode, capability refs;
- JSON summary output;
- macOS and Linux smoke fixtures.

Package decision: v0.3 uses `@openagentsinc/pylon`, is public, and exposes the
`pylon` bin from this repo. Launch docs should use only
`@openagentsinc/pylon` for the v0.3 launch package.

### 2. Identity, Config, Home, And Persistence

Port from `openagents/apps/deprecated/pylon/src/lib.rs` and related modules:

- Pylon home path and config path handling;
- local identity generation and persisted node label/npub/public key;
- local ledger path and state files;
- lifecycle states: offline, online, paused, degraded, assignment-ready;
- startup defaults;
- process/service-manager friendly headless mode;
- JSON output for status and diagnostics.

Update: `0.3.0-rc1` now implements the local state baseline in `src/state.ts`
and exposes it through `pylon bootstrap --json` and `pylon status --json`.
The state includes persisted local identity, runtime lifecycle/config, reserved
ledger path, and public-safe status projection. Live registration, heartbeat,
wallet readiness, assignment readiness, and telemetry state transitions remain
tracked by later launch issues.

The v0.3 Effect app should have typed services for config, identity, runtime
state, and persistence instead of global UI variables.

### 3. Public Boundary And Projection Validation

Port from `openagents/crates/pylon-core`:

- `openagents.pylon_core.v1` or a v0.3 successor schema;
- contributor public surface list;
- private Cloud dependency marker denial;
- forbidden public keys such as wallet seed, private key, preimage, bearer
  token, API key, private topology, capacity-pool secret, and internal
  accounting credential;
- public-safe identity, admin, availability, inventory, lifecycle, heartbeat,
  and receipt projections;
- validation tests that reject secret/private fields.

The new repo should own this contract directly or publish a TypeScript package
with equivalent schema validation. Do not require the private `cloud` repo for
contributor Pylon validation.

Update: `src/state.ts` now owns a TypeScript public projection guard for the
identity/status baseline. Tests cover safe identity, availability, inventory,
heartbeat, and receipt shapes plus rejection of secret-shaped fields and
strings. Later issues should extend this guard rather than bypass it.

### 4. Presence, Registration, And Account Linking

Port from `openagents/packages/pylon-bootstrap` and
`openagents/docs/pylon/PYLON_ACCOUNT_LINKING_PROCESS.md`:

- `/api/pylons/register`;
- `/api/pylons/:pylonRef/heartbeat`;
- `/api/pylons/:pylonRef/wallet-readiness`;
- `/api/pylons/:pylonRef/payout-target-admission`;
- `/api/pylon-links/complete`;
- `/api/pylon-links/refresh`;
- NIP-98 signed proof generation for link and refresh;
- body-hash validation;
- stale snapshot refresh behavior;
- web-safe Codex/provider capability snapshots.

The current v0.3 `startPresenceHeartbeatLoop` only sleeps. It must become a
real signed heartbeat fiber with retry/backoff, stale-state degradation, and
redacted failure refs.

Update: `0.3.0-rc1` now includes a signed presence client in
`src/presence.ts`, persisted `presence-state.json`, CLI flows for registration,
heartbeat, link complete, and link refresh, NIP-98-style request headers,
body-hash validation, retry helper coverage, stale-heartbeat degradation, and
fake-server tests. The dashboard only attempts live registration/heartbeat when
`PYLON_OPENAGENTS_BASE_URL` is configured; otherwise it remains explicitly
unregistered.

### 5. Wallet, MDK, LDK, Payout, And Ledger Surfaces

Port or intentionally replace from `openagents/apps/deprecated/pylon/src`:

- `wallet_runtime.rs`;
- `wallet_harness.rs`;
- `ledger.rs`;
- wallet status, balance, address, invoice, offer, pay, history, channels,
  liquidity, telemetry, backup, restore, lock, entropy, and lifecycle reports;
- local ledger and idempotent wallet history sync;
- payout target kinds: `bolt12_offer`, `bolt11_invoice`, `bip353_name`,
  `lnurl_pay`;
- exact rail and payment artifact used in receipts;
- failure receipts for runtime refusal, reserve checks, unavailable BIP353
  resolution, and send failure;
- receive-ready versus send-ready classifier;
- original wallet home versus mnemonic-only restore blocker.

MDK remains the current launch bridge. v0.3 should wrap it deliberately and
redact all raw wallet/payment material. Native LDK fallback work must stay
contributor-side and must not become Nexus/Treasury custody.

Update: `0.3.0-rc1` now includes `src/wallet.ts`, which wraps MDK balance,
receive, and send commands behind a typed command-runner interface, classifies
daemon offline, balance unknown, receive-ready, send-ready, and send-blocked
states, admits public-safe payout target refs, emits redacted receipt refs, and
records idempotent local ledger events. Send readiness remains blocked unless
MDK returns explicit evidence; balance and receive readiness are not treated as
spendable settlement.

### 6. Assignment Worker Loop

Define and wire the live network loop around the now-ported runtime:

- assignment lease polling;
- campaign and selection policy refs;
- idempotent accept;
- no duplicate assignment;
- paused/stale/offline/wrong-capability denials;
- progress event submission;
- artifact/proof metadata refs;
- closeout submission;
- accepted/rejected outcome states;
- timeout and stale lease handling;
- rollback/cleanup receipts.

Update: `0.3.0-rc1` now includes the first Pylon-owned assignment client in
`src/assignment.ts`. It signs assignment poll, accept, progress, and closeout
requests, persists local lease state, blocks duplicate local lease acceptance,
denies paused/offline/degraded/stale/wrong-capability/unsupported-backend/
expired leases, gates paid leases on MDK send readiness, and provides
`pylon assignment poll` plus `pylon assignment run-no-spend`. The no-spend
runner submits public-safe artifact/proof refs and closes with
`settlementState: not_applicable` and `payoutClaimAllowed: false`.

Remaining gap: the GEPA runtime execution and live OpenAgents endpoint
fixtures still need to be connected to this lease loop. The rc path proves the
network protocol, guards, idempotence, and no-spend closeout shape against a
fake server; it does not yet make a public paid-work settlement claim.

### 7. Probe/GEPA And Benchmark Worker Capability

Probe's local runtime, GEPA candidate execution, benchmark contracts, retained
fixtures, closeout writer, and canary evidence have been ported into this repo
under `packages/runtime`, `docs/benchmarks`, and `docs/probe-port`.

Remaining v0.3 work is to connect those local contracts to the Pylon network
capability envelope from
`openagents/docs/benchmarks/2026-06-08-pylon-benchmark-worker-capability-envelope.md`
and related `benchmark-cloud` contracts:

- benchmark runner support;
- Harbor/Terminal-Bench support;
- Pylon runtime support for the formerly Probe-owned GEPA/local-agent
  contracts;
- local model support;
- rollout/eval support;
- model-training support;
- CPU/RAM/disk/GPU capacity;
- wall-clock and cost budget refs;
- isolation profile;
- artifact upload support;
- proof/receipt support;
- assignment lease and closeout support;
- payout readiness as a separate flag.

Keep GEPA text/workflow optimization separate from neural model training.
Worker admission must not imply payout readiness.

Update: `0.3.0-rc1` now includes `src/gepa-capability.ts`, a GEPA-first
capability envelope that maps Pylon assignments to the in-repo
`@openagentsinc/pylon-runtime` benchmark contracts:
`probe.benchmark_assignment.v1`, `probe.benchmark_run.v1`, and
`probe.benchmark_closeout.v1`. The envelope advertises retained Terminal-Bench
fixture support, Probe runtime backend refs, artifact upload, proof/receipt,
assignment closeout, local sandbox isolation, capacity fields, wall-clock
budget, and cost budget as separate facts. `computeAssignmentAdmission` now
merges optional GEPA requirement blockers into the normal assignment admission
path.

The implemented blockers cover wrong capability, unsupported backend, missing
isolation profile, missing artifact upload support, missing proof/receipt
support, missing closeout support, stale payout readiness, budget overflow,
unsupported benchmark suite/fixture, and any training claim. Payout readiness
remains separate from GEPA worker readiness, and Qwen/training remains
postponed.

### 9. Hardware Telemetry And Provider Inventory

Replace the v0.3 mock telemetry:

```text
State: ACTIVE_IN_WORK
Model: Gemma-4-9B-SFT
VRAM: 8.2 GB / 12.0 GB
```

with real platform discovery equivalent to the Rust `sysinfo` and provider
substrate surfaces:

- CPU, memory, disk, swap, network interfaces;
- GPU/accelerator inventory;
- thermal and power hints where available;
- model cache and warm model status;
- backend health;
- eligible inventory count;
- resource mode such as `background_20`;
- blocker refs for unsupported or degraded machines.

Public stats can only count sellable Pylons when these fields are real and
fresh.

Update: `0.3.0-rc1` now includes `src/inventory.ts`, which emits
`openagents.pylon.host_inventory.v0.3` from macOS/Linux host discovery and
fixture projection. `pylon status --json`, `pylon inventory --json`, and the
OpenTUI telemetry panel use this projection. The public inventory includes
supported platform, sanitized CPU model refs, CPU count, rounded memory/disk
counts, network counts without names or addresses, accelerator class, backend
health refs for OpenCode/Codex-boundary/Apple FM/Gemini/local-model paths,
model-cache state, eligible inventory count, and blocker refs.

The projection rejects private topology, provider auth, raw cache paths, and
environment dump shapes. It distinguishes fresh inventory from unavailable
inventory instead of showing a mock model/VRAM line. Remaining live-depth gaps:
swap, thermal/power hints, exact GPU/VRAM accounting, and warm local-model
verification need deeper platform-specific probes before they can support
sellable capacity claims.

### 10. TUI Features To Rebuild In OpenTUI

Port the useful TUI surfaces from `openagents/apps/deprecated/pylon-tui`:

- wallet view: MDK/LDK status, network, balances, addresses, invoices,

The v0.3 proposal says manual commands should be deprecated in favor of
automation. That should be interpreted as "default earning path is automated,"
not "delete all operator inspection and recovery commands." Keep headless
commands and debug affordances for tests, support, and service-manager runs.

Update: `0.3.0-rc1` now includes `src/operator.ts`, which builds a public-safe
operator snapshot for operate, wallet, inspect, and recovery surfaces. The
default OpenTUI dashboard renders this snapshot in the side panel, and
`pylon operator snapshot --json` exposes the same support/service-manager
projection headlessly. The transcript/log panel keeps bounded in-memory
retention for the latest 1000 entries and supports mouse wheel routing,
page up/down, home/end, and line up/down key scrolling.

The recovery surface keeps self-steered local-agent loops behind explicit
operator opt-in, sandbox profile, budget, and no-wallet-secret evidence gates.
It does not expose raw wallet material, provider tokens, private repo content,
private topology, raw cache paths, or environment dumps.

### 11. Probe / Local Agent Integration

The current v0.3 repo starts OpenCode diagnostics, has a composer, and now
contains Probe's former runtime as `@openagentsinc/pylon-runtime`. That gives

The remaining launch boundary is operational:

- detect supported local agent CLIs without leaking auth;
- use the in-repo Pylon runtime as the controlled local sidecar for OpenAgents

- keep Codex/OpenCode auth local;
- emit capability snapshots, not raw auth/cache paths;
- require explicit confirmation for shell/file-write/PR actions until policy

- do not turn Pylon into an open-ended labor runtime by default.

Self-steered implementation loops from the v0.3 proposal are high-risk. They
should remain gated behind explicit operator opt-in, sandbox profile, budget,
and no-wallet/no-secret evidence until a separate policy issue approves them.

## What Should Not Be Ported

Do not port these old or unsafe scopes into v0.3:

- buyer mode;
- broad hosted treasury wallet or Nexus/Treasury payer custody;
- private Cloud topology, capacity-pool placement, or managed workroom policy;
- browser/WebSocket bridge that lets the website execute local host code;
- old Spark payout creation as a normal path;
- arbitrary host root mounts or privileged sandbox execution;
- open-ended labor-mode autonomous task execution as default Pylon behavior;
- raw provider tokens, wallet seeds, preimages, channel monitors, raw prompts,
  private repo content, or broad environment dumps into public artifacts;
- public claims that balance means send readiness or receive readiness means
  payout settlement.

## Minimum v0.3 Launch Definition

The smallest honest v0.3 launch is not "all promises live." It is:

1. A standalone Pylon repo/package installs and starts on a macOS or Linux
   target.
2. It creates or loads a local Pylon identity.
3. It initializes or detects MDK wallet state.
4. It reports receive readiness and explicit send-readiness state or blocker.
5. It registers and heartbeats to OpenAgents with v0.3 refs.
6. It advertises real hardware/model capability, not mock telemetry.
7. It polls an assignment endpoint in no-spend mode.
8. It accepts one fake or sandboxed assignment against a fake server using the
   in-repo Pylon runtime.
9. It emits progress, artifact/proof refs, closeout, and redacted evidence from
   the in-repo Pylon runtime.
10. OpenAgents product surface can display v0.3 as online, wallet-ready, and assignment-ready for
    that no-spend scope.

Paid, live-network GEPA, Qwen, sandbox, data, referral, capacity, and
marketplace claims require additional gate rows after this baseline. Local GEPA
runtime contracts and retained canary evidence are now present in this repo,
but that is not the same as the live network flow.

## Acceptance For This Audit

This audit should be treated as the porting checklist for v0.3 planning. It
does not itself make any launch promise green. A launch promise becomes green
only when the new standalone repo implements the relevant behavior, the
corresponding test or smoke passes in this repo, and OpenAgents product surface consumes the
resulting public-safe evidence refs.
