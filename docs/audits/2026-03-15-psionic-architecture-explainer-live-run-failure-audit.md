## Exact Failure Sequence

### 1. First live run failed on packing policy

The first end-to-end attempt failed before training could start:

```text
Failed to build Apple adapter backend: sequence `sample-000005` uses 157 tokens but packing policy only allows 96
```

This showed that the operator lane still had a stale hard-coded `96` token
packing window while the frozen architecture-explainer corpus now contains a
longer structured sample.

### 2. I changed the operator lane to derive packing from the frozen corpus

I updated the in-progress local code to derive `max_row_tokens` and
`max_batch_tokens` from explicit token captures instead of hard-coding `96`.

The next run logged:

```text
launch: derived packing window=272 tokens batch_budget=544 tokens from frozen corpus
```

That fixed the first blocker.

### 3. The bridge then surfaced the Apple Intelligence "needs an update" popup

While retrying the live run, macOS surfaced:

```text
This app needs an update in order to work with the latest Apple Intelligence models.
Some features may not work as expected.
```

Important detail:

- that text does not exist anywhere in this repo
- the popup was generated by macOS / the Apple runtime, not by our Rust or
  Swift code

The most likely reason is that the sidecar was being launched as a bare
ad-hoc-signed executable from `bin/foundation-bridge`, with no proper app-bundle
metadata.

### 4. I tried a local packaging fix for the bridge helper

In local uncommitted work, I changed the Swift bridge build path so it also
produces a signed helper bundle:

- `bin/FoundationBridge.app/Contents/MacOS/foundation-bridge`

and updated local bridge discovery to prefer that helper.

After rebuilding, the helper health response changed from the old minimal
payload to bundled metadata:

```json
{
  "status": "ok",
  "model_available": true,
  "version": "1.0.0+20260315230711",
  "platform": "macOS 26.1.0"
}
```

That was a real improvement in helper packaging, but it did not fix the
operator run by itself.

### 5. The next live run failed on base-model signature mismatch

The first staged package used the wrong base signature:

- staged package: `49ef0b96ba2c39574f768900169a1ed6ba13a3d1`
- live compatible signature: `9799725ff8e851184037110b422d891ad3b92ec1`

The bridge rejected the package as incompatible.

I then changed local in-progress code so the reference-run harness pinned the
experiment manifest's real Apple compatibility signature into the operator
launch instead of relying on the old synthetic hash-only compatibility anchor.

The next run correctly logged:

```text
launch: derived Apple runtime lineage model=apple-foundation-model use_case=general guardrails=default base_signature=9799725ff8e851184037110b422d891ad3b92ec1
launch: pinned Apple compatibility signature 9799725ff8e851184037110b422d891ad3b92ec1 from operator contract
```

That fixed the second blocker.

### 6. The current hard blocker: Apple rejects the staged package as an invalid asset

After the signature fix, the bridge still refused the staged package.

The exact bridge error is:

```json
{
  "error": {
    "code": "adapter_incompatible",
    "failure_reason": "invalid_asset",
    "message": "Adapter 'debug-architecture-explainer-latest' is not a valid Foundation Models asset",
    "debug_description": "Failed to compile draft model."
  }
}
```

This is the real blocker now.

## Why I Was Not Actually Using A Valid Psionic Apple Export

The current Psionic export path in `crates/psionic/psionic-train/src/apple_adapter.rs`
does not emit the same kind of artifact the Apple runtime accepts.

What the current Psionic path does:

- trains the repo-owned narrow reference backend
- exports the final model state via `PortableModelBundle::export_safetensors()`
- writes those bytes directly into `.fmadapter/adapter_weights.bin`

What the Apple toolkit does instead:

- converts a real toolkit/TAMM checkpoint into Apple's expected adapter blob
  format using `AdapterConverter`
- writes the resulting weights through Core ML blob storage

The practical evidence is very clear.

Known-good toolkit export:

- `/tmp/openagents-apple-adapter-run/export/mutex_helper.fmadapter/adapter_weights.bin`
- size: `133312320` bytes
- bridge load result: success

Current Psionic-staged export:

- `~/.openagents/logs/autopilot/apple-adapter-training/.../staged/psionic-architecture-explainer-first-real-run.fmadapter/adapter_weights.bin`
- size: `16744` bytes
- bridge load result: `invalid_asset`

That size gap is not a cosmetic difference.
It is the direct evidence that the current Psionic export is not writing the
Apple-valid adapter payload format.

## What I Think We Should Do Next

### 1. Stop pretending the current Psionic export is Apple-valid

This is the first thing to fix in docs and issue closure discipline.

Right now the bridge says the package is invalid.
That should be treated as authoritative.

### 2. Split the problem into two explicit lanes

Lane A: repo-native Psionic reference training

- keep this for substrate development, metrics, receipts, and experiment logic
- do not describe it as Apple-valid runtime export until it actually is

Lane B: Apple-valid toolkit-backed live adapter path

- use the Apple toolkit when the requirement is a real loadable `.fmadapter`
- make this explicit in operator docs and issue comments

### 3. Add a real toolkit-backed operator path in the desktop control layer

If the goal is to complete the real first run soon, the fastest honest path is:

- convert the frozen OpenAgents dataset into a toolkit-fed train/eval run
- run toolkit training with operator-controlled memory settings
- export with `export.export_fmadapter`
- then continue with held-out eval, bridge smoke, and authority acceptance

This belongs in the app-owned operator layer, not hidden behind manual shell
commands.

### 4. Keep the Psionic work, but rename its status honestly

The current Psionic Apple lane should be described as:

- a repo-owned reference training and evaluation backend
- not yet a proven Apple-valid asset exporter

That is still useful work. It just is not the same thing as a completed Apple
adapter production lane.

### 5. Re-run the first real issue only after one of these is true

Only retry `#3658` after one of the following is implemented:

- Psionic emits an adapter payload that the bridge accepts as a valid asset, or
- the operator flow has an explicit toolkit-backed export path that produces a
  bridge-loadable package

Without one of those, more retries will only repeat the same failure.

## Bottom Line

I did not stop using Psionic because Python was more convenient.
I started testing the Python toolkit because the live Apple runtime proved that
the current Psionic `.fmadapter` export is not a valid Foundation Models asset.

The right next move is not more issue closure theater.
The right next move is to choose one honest path:

- either finish a real Apple-valid export path from Psionic, or
- explicitly integrate the Apple toolkit as the live export lane for the first
  real operator run

Until then, the first real `Psionic architecture explainer` run is still
incomplete.
