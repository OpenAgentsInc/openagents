## Why rc.2, not the admitted rc.1

FA-REL-01 admitted a signed `0.1.1-rc.1` candidate at source `24b7622432`
(artifact SHA-256 `05f449b6…`), but held the publish as one unit. That
artifact was never uploaded to Cloud Storage or GitHub. Its local build
directory no longer exists. Only the staged manifest, the signature, and the
recorded hashes survive. A version string must bind to exactly one artifact
identity, so the `0.1.1-rc.1` identity is retired unused and this publish cuts
a fresh signed `0.1.1-rc.2` from current `origin/main`. Current main contains
every commit in the admitted candidate plus the automatic update scheduler
(`8bebf98426`) and later fixes, and the rc.1 DMG filename residual (a DMG
labeled `0.1.0-stable`) does not recur: the fresh artifact name carries the
correct version.

## Feed-path boundary (recorded honestly)

The desktop client fetches the ReleaseSet v2 feed first and uses the v1
manifest only when the v2 feed is unavailable (bounded macOS arm64
compatibility until `2026-10-14`). The rc v2 release-set still serves
`0.1.0-rc.25`, because a v2 promotion requires all four targets
(darwin-arm64/x64, linux-arm64/x64 — ten artifacts) through the release
coordinator, and only the darwin-arm64 target was rebuilt here. So:

- v1-era rc installs (≤ rc.13) see `0.1.1-rc.2` on their feed path now.
- v2-era rc installs (rc.25) still see `0.1.0-rc.25` until a four-target
  `0.1.1-rc.x` set is promoted per
  `docs/deploy/2026-07-21-atomic-coordinator-rc-promotion-and-tailnet-linux-acceptance-runbook.md`.
- The first build whose updater checks automatically is `0.1.1-rc.2` itself
  (the scheduler landed after rc.25), so unattended-update observation starts
  with installs of this version or later.
