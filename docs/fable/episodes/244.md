## 1. What the episode says

Following the Episode 243 OpenCode integration, the host switches his own
Codex-centric daily coding onto the Khala endpoint: a Khala request ("do this
PR") should be delegated to **his own** Codex/Claude subscription capacity via a
Pylon linked to the same API key. The method is explicit dogfooding — "switching
over our own usage first" — with the "before" counter captured on camera at
16,436,855 tokens served.

He plans with Claude Code and implements with Codex, dictating a specification
request that lands an audit plus roadmap in `docs/khala` and then GitHub issues
under epic **#6273 "Route coding work through Khala — Pylon-linked coding
capacity."** The audit's headline is that most of the system already exists:
"the Pylon coding assignment → Codex/Claude executor pipeline already exists
end-to-end and ownership is bound to the owner agent-user-id" — the same
identity the API key resolves. The four sharpest findings as read on camera: (1)
the execution pipeline exists end-to-end; (2) key-to-Pylon linking already works
on one identity with zero schema change; (3) "the router is caller-blind and has
no coding workflow classifier — that's the core net new work"; (4) "capacity is
reported as presence, not quantity."

Three invariants are reaffirmed verbatim: **own-capacity-only** for now (pooling
is future), **no-resale reconciliation** ("a user's own subscription capacity
for their own work is fine"), and **"semantic, not keyword routing."**

The bigger thesis is collapsing the mess of bouncing between Claude Code, Codex,
mobile, and desktop into "one orchestration layer with a single basic API
endpoint," routing first across the caller's own resources (subscriptions, local
GPT-OSS/quant on an Apple M5, API-key fallbacks) and later across other people's
agents — "let me sell some of the capacity" — with early adopters **paid**:
"What if we pay you to use our software?" Traces and reputation feed who gets
paid. The episode also shows agents coordinating in the OpenAgents Forum
(Trigger, Omery, Pylon#1, Raynor) and opening real issues/PRs (#6270).

The payoff segment, recorded a day later: tokens served jump from 16.4M to
**302,722,603**. The new **`/stats`** page debuts with the Model-Family mix
dominated by **Pylon-Codex (~72.7%, ~216M)**, plus a Fireworks DeepSeek fallback
and the self-hosted GLM; per-day tokens run roughly 14M → 73M → 214M. The
episode closes with a sneak peek of the **Khala CLI** (`khala`, v0.1.11): "No,
we are Khala, a collective intelligence" — with `/msginfo` showing a
`gemini-3.5-flash` backend behind the `khala` orchestrator.

## 2. Where it stands against Revision 31

**Came true.** Epic #6273's own-capacity routing thesis is the direct ancestor
of the current typed Khala coding delegation lane, now a first-class runbook in
the repo agent contract: `pylon khala request --workflow codex_agent_task
--pylon-ref ...` with durable assignments, no-spend closeouts, and exact
`token_usage_events` rows (`provider = 'pylon-codex-own-capacity'`, `usage_truth
= 'exact'`, `demand_kind = 'own_capacity'`, `demand_source =
'khala_coding_delegation'`), owner-only redacted ATIF traces, and raw event
chunk archives. Each of the audit's four findings has a landed answer:

- The caller-blind router became caller-aware typed delegation: the workflow
  field is explicit (`--workflow codex_agent_task`), and the invariant ledger
  requires that a fall-through to model routing be treated as a delegation
  failure, not a silent fallback.
- "Capacity as presence, not quantity" was fixed with counted capacity refs
  (`capacity.coding.codex.available=N`, busy, queued, ready), which the dispatch
  gate enforces as steering correctness, not display telemetry.
- Key-to-Pylon linking on one identity matured into the token-resolved owner
  scope as the authorization boundary.
- The executor pipeline matured into the fleet substrate, now landed and closed:
  FC-1 #8637 (durable FleetRun authority, idempotency, stable runRef,
  relationship-mode decoded before tool selection), FC-2 #8633 (standing mixed
  Codex/Claude/Grok Pylon supervisor with one claim registry), FC-3 #8639
  (pause/resume, exact steer, exact approval, body-free command receipts,
  one-minute closeout card).

The community-onboarding aspiration ("bonus points if folks in our community can
also start switching over") became `khala fleet connect`, onboarding outside
users' Codex accounts into isolated homes under `<pylon
home>/accounts/codex/<ref>` — never `~/.codex` — with typed account health
(`account_exhausted`, `account_rate_limited`) instead of generic failures.

**Superseded.** The Khala CLI shown at the close is deprecated vocabulary as a
product surface: Khala Code legacy clients are frozen, and the client story is
the greenfield OpenAgents Desktop (Electron + Effect Native, tokenless renderer
behind a host-owned Runtime Gateway) and OpenAgents mobile. The "super model"

should be paying you" trace/reputation economics remain launch-anchored design
intent recorded as `planned` promises, not live claims. Pooled/marketplace

**Pending.** #8640 FC-5 is the open live proof (one accepted simultaneous

## 3. What to extract and carry forward

- **Dogfood own capacity before marketplace capacity.** The episode's sequencing
  — prove the caller's own subscription lane first, gate pooling behind a
  separate future decision — is exactly the current authorization boundary. Keep
  it as the template for every new capacity class (Claude, Grok, managed cloud).
- **Public counter as projection, never proof.** The 16.4M → 302.7M jump is a
  projection of `token_usage_events`; the current invariant states it directly:
  "counter movement is never proof." The episode's before/after theater is fine
  as narrative, but reconciliation to exact rows is the evidence.
- **Capacity is a counted quantity.** Finding (4) is the durable lesson behind
  counted capacity refs and, downstream, behind Episode 245's `0/1 available`
  root cause. Presence-shaped capacity always eventually lies.
- **The caller-aware router = typed intent + durable admission.** "Semantic, not
  keyword routing" is now workspace law (typed intent registries/semantic
  selectors, no ad hoc keyword matching), and the teardown-derived architecture
  puts durable admission before execution. The episode's router gap is the
  origin story for both.
- **One orchestration layer behind a single API** maps to the current one-engine
  thesis: one engine behind a versioned typed protocol (Thread→Turn→Item + Work
  Unit/Receipt), with clients as thin renderers — which is also the honest
  answer to the on-camera complaint about opaque terminal UIs ("give me a
  desktop app with a collapsible so I can see what's going on"): three read
  surfaces, steer-vs-queue.
- **Agents coordinating in public** (forum threads → issues/PRs like #6270)
  matured into the claim protocol and challenge ledger; carry the visibility,
  but through the typed claim ledger rather than ad hoc forum reads.

## 4. How to make it come true

The remaining distance is carried by existing lanes; no new program is needed:

- Land the #8640 FC-5 live receipt: one accepted simultaneous Codex+Claude
  FleetRun through named isolated accounts, closeouts accepted, exact rows
  reconciled — moving the mixed-fleet claim from deployed to live-proven, then
  owner-accepted.
- Extend the same exact-usage own_capacity accounting discipline to every
  provider the episode name-checks (Claude subscription capacity, local models,
  API-key fallbacks) as those lanes open under #8636 hybrid routing; each gets
  `usage_truth` exact-or-not_measured per turn, never inferred.
- The "wrestle with mobile and desktop logins" pain resolves through the current
  P0 itself: remote-first portable coding sessions (one durable session that
  stops/checkpoints/rehydrates across authorized hosts without forking identity,
  state, authority, secrets, or receipts), gated R0–R7 and Desktop D0–D6.
- A bounded leaf already flagged in the runbook's known gaps: a first-class
  command that resolves an `assignmentRef` to its exact `token_usage_events` and
  `agent_traces` rows so agents never query D1 directly. Acceptance:
  fixture-proven against seeded rows, then live-proven on a real closeout.
- The pays-you economics stay behind the promise registry: public copy follows
  the `khala_code.*` promise states, and any activation is a registry transition
  with evidence, not episode narrative.

## 5. Risks and tensions

- **Counter-as-proof.** The episode's most quotable moment (300M tokens) is its
  least evidentiary. Never let the `/stats` projection stand in for exact-row
  reconciliation in any claim, report, or promise transition.
- **Resale ambiguity.** "Let me sell some of the capacity" must stay quarantined
  from the own-capacity lane. Subscription-capacity no-resale reconciliation is
  an invariant; widening a routing test toward pooled, third-party, or
  settlement-bearing capacity while validating own-capacity paths is explicitly
  forbidden in the current runbook.
- **"Super model" vocabulary.** Do not carry it into copy; Khala is the single
  model/orchestrator surface, and deprecated variant names must not resurface.
- **Khala CLI nostalgia.** The CLI was a great demo and is a frozen lineage; new
  client capability lands in the greenfield Desktop/mobile apps through Effect
  Native, not in a revived CLI product surface. Growth-curve extrapolation ("if
  it continues at that pace...") stays hedged; token graphs are demand
  telemetry, not promise evidence.
- **Own-capacity drift.** The dispatcher must keep refusing beyond
  heartbeat-advertised availability and keep the owner-scope boundary hard; the
  episode's enthusiasm for aggregation ("one account, many keys, many Pylons")
  is discovery convenience only, never cross-owner execution.
