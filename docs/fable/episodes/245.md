## 1. What the episode says

The host switches his operator workflow from "Codex hitting the Khala API to
steer the Codex Fleet" to a new Khala Code desktop app (an "electrobun effect"
shell launched by typing `khala`) doing the same steering. The `/stats`
dashboard shows over 6.5B tokens served with a 2.48B single-day peak (June 29),
and the stated goal is saturation: "I want to fan out any number of Codex... I
want to fully saturate it."

The failure that drives the episode: the desktop app's fleet tools
(`pylon_ensure`, `codex_fleet_status`, `codex_spawn`) cannot reliably spawn
workers. On camera, `codex_spawn` fails with "No Pylon Codex assignment capacity
is available right now (0/1 available)." The root cause, produced by a live
investigation and written into the GEPA usage doc: the failure was "not because
the fleet was truly full, but because the desktop tool never advertises
capacity" — `codex_spawn` "runs a partial sequence" and "skips the one module
that raises capacity, so when the single default slot is busy, it dead-ends
instead of recovering." The host recognizes it as "the exact same brittleness I
hit manually all session": a missing or misordered deterministic step with no
fallback.

The prescribed fix is a typed program signature, **`khala.fleet.delegate`**: a
fixed, hand-written module pipeline — `ensure_pylon` → `advertise_capacity` →
`select_account` → `prepare_work` → `dispatch` → `verify_closeout` — where "each
module has a precondition and a deterministic fallback to the module that
satisfies it." Dispatch's missing-capacity case loops back to
advertise-capacity, so "the bundle cannot dead-end on 0/1." The load-bearing
sentence: "The control flow is code, not an LLM decision per call."

GEPA (the DSPy optimizer) is positioned around, not inside, that program: "GEPA
does not decide the steps. That would make it less deterministic, the opposite
of what you want." The frame is DSPy's — a deterministic program whose
textual/policy parameters (slot counts, capacity-advertisement policy given N
ready accounts) are compiled offline by GEPA reading rich redacted delegation
traces. The architecture is hybrid: **Mutalisk**, a new standalone Python
DSPy/GEPA offline-optimization repo (sibling to the Hydralisk self-hosted
inference lane), runs the offline search, under the rule "Python is offline/
leaf compute that produces untrusted candidates and evidence; Effect is the
online authority that selects, gates, and admits them. A GEPA win is a
candidate, never an auto-promotion." The episode closes by filing epic **#7730
"Eliminate delegation dead-ends — deterministic delegation program + GEPA
loop"** with child lanes (GD-P #7731: the `khala.fleet.delegate` program,
modules, and typed error taxonomy; plus regression-lock and GEPA lanes across
OpenAgents and Mutalisk) and tasking them to Codex.

## 3. What to extract and carry forward

- **Deterministic handoff over model-improvised spawn.** The episode's core
  exchange — replace an LLM choosing tools per call with a typed program whose
  control flow is code — is exactly durable admission before execution plus the
  typed intent registry. It also aligns with workspace law: no ad hoc keyword
  matching for routing; typed selectors decide, deterministic parsing only after
  selection.
- **Every precondition gets a satisfying module.** "0/1 available" was a missing
  precondition wearing a capacity costume. The durable rule: failure classes
  must be typed and recoverable (loop back to the module that satisfies the
  precondition), never generic dead ends. Current form: typed account health,
  counted capacity refs, `unknown_pending_reconcile`.
- **Optimizer as candidate generator, never authority.** "A GEPA win is a
  candidate, never an auto-promotion" is worth keeping verbatim even with GEPA
  itself pruned. It is the same governance shape as the six proof rungs and the
  challenge ledger: untrusted evidence in, gated admission out. Any future
  self-improvement lane must inherit this sentence.
- **Offline compute / online authority split.** Leaf compute produces untrusted
  candidates and evidence; the typed online authority selects, gates, and
  admits. This survives independently of Python, Mutalisk, or Workers.
- **Rich redacted traces make delegation optimizable — and auditable first.**
  "We already produce rich redacted traces for every Codex delegation" matters
  today for audit and closeout proof, and only later (if ever) for
  optimization.

- The deterministic contract is carried by the closed FC-1..FC-3 lanes and the
  Khala→Pylon→Codex runbook; the next receipt is #8640 FC-5's accepted
  simultaneous Codex+Claude FleetRun with reconciled closeouts, moving the
  mixed-fleet claim to live-proven and then owner-accepted.
- The episode's "single user prompt → deterministic program" UX belongs in the
  greenfield Desktop under D0–D6 and the R0–R7 remote-first session gates: the
  fleet surface renders FleetRun state through the typed protocol
  (Thread→Turn→Item + Work Unit/Receipt) with steer-vs-queue delivery — not a
  revived electrobun client or `codex_spawn`-style tool improvisation.
- Regression-lock the extinct failure class where not already covered: a
  bounded leaf adding an oracle that a dispatch refused for missing advertised
  capacity yields the typed capacity class (never a generic "0/1" string) and
  that the gate admits at most heartbeat-advertised slots. Acceptance:
  fixture-proven in the normal sweep, then live-proven on a real
  refused-then-recovered dispatch.
- **GEPA residue: a post-R7 revisit gate, not a revived program.** If
  delegation-parameter tuning (slot policy, account selection) shows measurable
  pain after the substrate is live-proven, open it as a challenge-ledger entry
  with a named metric and a candidate/admission boundary consistent with
  "never an auto-promotion." Do not re-create Mutalisk, a Python lane, or any
  optimizer with dispatch authority ahead of that gate.
- The teaser's pays-you economics advance only as `khala_code.*` (or successor)
  promise transitions with evidence; the transcript-file mismatch (245.md vs

## 5. Risks and tensions

- **Optimizer loops owning dispatch decisions** is the explicit anti-pattern,
  named inside the episode itself. Any resurrection that lets a search loop
  mutate control flow, auto-promote parameters, or gate admission violates both
  the episode's own design and the current governance rungs.
- **Reviving pruned lanes by momentum.** Mutalisk/DSPy/GEPA read as attractive
  infrastructure; they are out of the active program. Citing this episode is not
  authority to rebuild them.
- **Prompt-only delegation imitations.** Re-wrapping `codex_spawn`-style LLM
  tool improvisation in better prompts reproduces the failure class the episode
  exists to kill. Delegation goes through the typed FleetRun/`pylon khala
  request` contract or it does not go.
- **The scripted completion segment as evidence.** The final section of
  `24X1.md` is an unrecorded draft (imagined closed issues, imagined UI). Never
  cite it as demo footage or completion proof.
- **Counter-as-proof.** 6.5B served and the 2.48B peak day are projections;
  "counter movement is never proof" applies with full force to saturation
  narratives.
- **Teaser economics as live claims.** "Your coding agent pays you" is
  `planned`-state design intent with an on-camera hedge; public copy that
  outruns the promise registry is a claim violation, not marketing. Likewise,
  the frozen Khala Code shell in the footage carries UI patterns forward only
  through Effect Native components and the behavior-contract registry, never
  by reviving the shell.
