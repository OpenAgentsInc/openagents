## 1. Current state — one paragraph (SUPERSEDED by the UPDATE above; original audit finding)

**No. GLM-5.2-REAP-504B is NOT serving real `openagents/khala` traffic right now, and a
real public `openagents/khala` request would NOT land on GLM in the committed production
configuration.** The full software path is built and wired end-to-end on the openagents
side (adapter, replica pool, singleflight, heartbeat, fail-closed arming, router-first
ordering, cost profiles, receipt disclosure, benchmark/profile schema, and a verification
smoke), and the Hydralisk side genuinely has **two Spot GLM endpoints up, loaded,
bearer-gated, HTTPS-reachable, and benchmarked at ~47 tok/s single-flight**. But the lane
is **fail-closed and unarmed in committed prod config**: there are **zero
`HYDRALISK_GLM_52_REAP_504B_*` entries in `wrangler.jsonc`**, so the GLM adapter is never
registered and dispatch skips it; arming is possible only via Wrangler secrets that are not
in the repo (an owner action). Compounding this, the committed
`KHALA_BACKING_MODEL=deepseek-v4-flash` would put **Fireworks DeepSeek first** even after
GLM is armed. The Khala→GLM end-to-end smoke (#6259) is a real verifier but **inert / no
PASS evidence** because the lane is unarmed. The Terminal-Bench replication (#6253) has a
**preliminary pilot only** (~70% solved/attempted, explicitly not decision-grade), run
**directly against the raw GLM endpoint, bypassing Khala** — there is **no Khala-routed GLM
Terminal-Bench run** yet.

**Arming gates (both must pass; both currently fail):**

## 3. The Hydralisk serving infra

**Model identity (PROVEN, repo-consistent, not Cerebras-served):** `0xSero/GLM-5.2-504B`
@ `cb6b1e0…12`, MIT, arch `GlmMoeDsaForCausalLM`; 78 layers (3 dense, 75 MoE, 1 MTP), 168
routed experts (keep-168), 8 experts/token, NVFP4 (`modelopt_fp4`), Router-KD recovery,
native context 1,048,576. Refs: `docs/glm-5.2-reap-504b-g4-runbook.md:45-64`,
`profiles/glm-5.2-reap-504b-b12x-g4.json:13-53`. A 469B fallback
(`0xSero/GLM-5.2-NVFP4-REAP-469B`) is documented but `not_started`.

**Where the infra lives:** GLM serving is **not** in the Python package (`hydralisk/admission/`
is all DeepSeek-V4). GLM is driven by **8 GCE/gcloud shell scripts** (`scripts/*glm-52-reap*`:
probe-admission, stage, launch-b12x, expose-private-proxy, expose-public-https,
install-durable-canary, run-terminal-bench-preflight), **1 fully-populated profile JSON**
(`profiles/glm-5.2-reap-504b-b12x-g4.json`, 26KB), and the **model-agnostic FastAPI proxy**
`hydralisk/serve/proxy.py` (singleflight 429 gate, bearer auth, `/health`, `/v1/models`,
`/v1/chat/completions`, `/hydralisk/v1/{capabilities,metrics,receipts}`, default-sampler
injection). GLM identity/aliases/sampler defaults are injected via `HYDRALISK_*` env set by
the launch scripts (`config.py:62, 102-106, 190-196`), not hardcoded.

**Host (PROVEN, with a hard caveat):** A clean standalone `g4-standard-192` (true 4× RTX PRO
6000) was **capacity-exhausted** (`ZONE_RESOURCE_POOL_EXHAUSTED_WITH_DETAILS`) on **both Spot
and Standard, in `us-central1-b` and `-f`** (profile:266-301). The runtime proof therefore runs
on **4 GPUs (0-3) selected inside an 8× `g4-standard-384` Spot fallback host**
(`…g4-8g-b-20260624214500`, internal IP `10.128.0.38`, no external IP, Spot/STOP, 1500GB
Hyperdisk, auto-delete false) — `receipt:61-82`, `runbook:96-108`.

**Serving stack (PROVEN):** vLLM via the **b12x SM120** distribution, pinned image
`voipmonitor/vllm@sha256:ce23a9b0…`, vLLM `0.11.2.dev279+…b12xpr11.cu132`. Launch
(`runbook:206-226`, `launch-glm-52-reap-504b-b12x-gce.sh:20-32`):
`--served-model-name glm-5.2-reap-504b-g4`, `127.0.0.1:8000`, `--tensor-parallel-size 4
--decode-context-parallel-size 4`, `--quantization modelopt_fp4 --kv-cache-dtype fp8`,
`--attention-backend B12X_MLA_SPARSE --moe-backend b12x`, `--tool-call-parser glm47
--reasoning-parser glm45`, `--max-model-len 250000 --max-num-seqs 2 --gpu-memory-utilization
0.95`, DSA indexer override, **MTP-2 speculative decoding** (`num_speculative_tokens=2`, default
ON in the launch script). Sampler guardrails: `min_p=0.05` (non-MTP only — vLLM speculative
decoding rejects `min_p`/`logit_bias`, so **`min_p` is omitted on MTP requests**),
`repetition_penalty=1.05` (1.10 loop bump), `enable_thinking=false`, `max_tokens` capped
(runbook:70-83, profile:64-71, 570-581). Context: **250K tested stable** (largest passed prompt
240,012 tokens); two concurrent full-250K requests **not** admitted (vLLM 1.94× at ceiling).

**Endpoint / reachability (PROVEN; #94 resolved):** Three layers on the same VM — raw vLLM
`127.0.0.1:8000` → bearer-gated private proxy `127.0.0.1:8080` (systemd
`hydralisk-glm52-reap-private-proxy.service`) → **Caddy v2.11.4 HTTPS front** reverse-proxying
only `/health`, `/v1/*`, `/hydralisk/*` (else 404). **Issue #94 was the hard blocker:** Khala is
a Cloudflare Worker that cannot reach the VPC-private `10.128.0.38:8080`. **Cloudflare Tunnel was
the first preference but the local CF token couldn't manage DNS/tunnel resources**, so the fix
fell back to a **GCE reserved static IP + tag-targeted 80/443 firewall + Caddy ACME TLS**
(`public-https-origin.md:41-58`). The concrete origin URL and bearer are deliberately untracked;
Worker arming is `HYDRALISK_GLM_52_REAP_504B_{BASE_URL,BEARER_TOKEN,ENABLED=ready}`
(`public-https-origin.md:139-150`).

**Two endpoints (PROVEN; matches #6262 pool):** Endpoint 1 = the 8× Spot fallback (GPUs 0-3).
Endpoint 2 = a **separate standalone Spot `g4-standard-192` (true 4×)**,
`…g4-4g-b-20260625154532`, brought up via a **cloned read-only model disk**, with its own
watchdog and HTTPS origin (`second-endpoint.md:49-60`). Hydralisk exposes the metadata; the pool
*routing* lives on the Khala side (#6262/#96).

**Measured (PROVEN):** load smoke TTFT 0.2376s / total 0.461s (receipt:120-128); endpoint-2
streaming median **TTFT 0.281s, 46.71 completion tok/s** (49.37 excl. TTFT) (second-endpoint.md:184-200);
startup ~498s weight load, KV cache 377,942 tokens, ~93GB/GPU resident. **8× TP is not a clear
win over 4× TP** (8 GPUs are PCIe-only, not one NVLink domain — khala-canary-status.md:188,199-214).
