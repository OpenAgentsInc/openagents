> Source: https://huggingface.co/nvidia/GLM-5.2-NVFP4 (HF release 2026-06-25, MIT license).

## Verdict (two questions)

1. **Is it worth adopting?** **Yes — strong candidate, likely both a quality upgrade AND a
   potential fix for our live tool-calling outage (#6310).** It is the **full** GLM-5.2
   (753B total / 40B active MoE), NVIDIA-quantized to NVFP4 with near-FP8 accuracy and
   **NVIDIA-validated agentic tool-use** — vs the **pruned** `0xSero/GLM-5.2-REAP-504B` we
   serve today, whose tool-calling is currently hard-down.
2. **Does it fit our GCP capacity?** **Yes — on our one 8-GPU host, not the 4-GPU ones.**
   It needs **TP-8 (~381 GB NVFP4 weights)**; our fleet is RTX PRO 6000 **Blackwell**
   (NVFP4-capable — we already serve NVFP4 on it). The single **`g4-standard-384` (8× RTX
   PRO 6000 = 768 GB)** host fits it; the nine **`g4-standard-192` (4× = 384 GB)** hosts do
   **not** (weights alone ≈ all VRAM, no KV headroom). Scaling beyond one host needs more
   8× Blackwell (us-central1 is at its 16-GPU quota → quota increase / other regions).

## What it is

- **Architecture:** `GlmMoeDsaForCausalLM` — GLM-5.2 MoE, sparse attention via the
  IndexShare/DSA indexer, **1M context**. **753B total, 40B activated.**
- **Quantization:** NVFP4 via NVIDIA Model Optimizer (v0.46.0). Only the MoE-expert linear
  weights/activations are quantized; **the shared expert is left unquantized.** ~381 GB
  on-disk weight footprint (≈753B × ~0.5 B/param at 4-bit).
- **License:** MIT (same as base) — commercial use OK.
- **Runtime:** SGLang + vLLM; **Blackwell (B200/B300)**; Linux; **requires
  `transformers>=5.3.0`** (the `glm_moe_dsa` arch).
- **Canonical launch (from the card):** `--tensor-parallel-size 8 --quantization
  modelopt_fp4 --tool-call-parser glm47 --reasoning-parser glm45 --trust-remote-code
  --chunked-prefill-size 131072 --mem-fraction-static 0.80`.

## Quality — NVFP4 ≈ FP8, and full ≫ our pruned checkpoint

NVIDIA's eval (NVFP4 vs the FP8 baseline of the same full model):

| Benchmark | FP8 baseline | NVFP4 |
|---|---|---|
| GPQA Diamond | 89.52 | 89.39 |
| SciCode (Code) | 49.85 | 49.04 |
| IFBench | 74.95 | **75.81** |
| AA-LCR (long-context recall) | 69.38 | **70.13** |
| τ²-Bench Telecom (agentic tool-use) | 97.9 | **98.25** |

NVFP4 is within noise of FP8 (better on 3/5). Crucially for us: **τ²-Bench Telecom 98.25**
(agentic dual-control tool-use) and the calibration set includes **Nemotron-SFT-Agentic-v2
(~992K tool-calling trajectories)** and **Nemotron-SFT-SWE-v2** — i.e. this checkpoint is
explicitly tuned/validated for **tool-calling and SWE-agent** work, which is exactly the
OpenCode workload our REAP checkpoint is failing.

## vs our current `0xSero/GLM-5.2-REAP-504B`

| | NVIDIA full NVFP4 | Our REAP-504B NVFP4 |
|---|---|---|
| Params | **753B / 40B active (full)** | 504B (REAP-pruned — experts dropped) |
| Quant | NVFP4 (NVIDIA ModelOpt, shared expert unquantized) | NVFP4 (keep-168 cut) |
| Parsers | `glm47` / `glm45` | **`glm47` / `glm45` (identical)** |
| Tool-use | NVIDIA-validated (τ²-Bench 98.25, agentic SFT) | **broken in prod (#6310: ~100% provider_error)** |
| License | MIT | MIT |
| Support | vendor (NVIDIA) launch recipe + ModelOpt | community |

**The #6310 link (important):** our REAP launch **already uses the same `glm47`/`glm45`
parsers** NVIDIA specifies — so #6310's `provider_error` on tool calls is **not** a
wrong-parser-value bug. That points the finger at **the REAP-pruned checkpoint itself**
(pruning likely damaged tool-calling) or a vLLM tool-path issue on it. The full,
agentic-validated NVIDIA model is therefore a **candidate fix**, not just an upgrade.

## Does it fit our GCP capacity (detail)

Our GLM fleet (10 replicas, RTX PRO 6000 Blackwell @ ~96 GB each):
- **1× `g4-standard-384` (8 GPU = 768 GB):** **FITS** the full 753B NVFP4 — ~381 GB weights
  (+ shared-expert unquantized + KV) inside 768 GB at TP-8, matching the card's `--tp 8`.
- **9× `g4-standard-192` (4 GPU = 384 GB):** **DOES NOT FIT** — 381 GB weights ≈ entire
  VRAM, leaving no room for KV/activations at usable context. These stay on the 504B REAP.
- NVFP4 datatype: ✅ supported (Blackwell; we already run `modelopt_fp4` here).
- Software: needs `transformers>=5.3.0` in the serving image (the card installs it in the
  SGLang container); both vLLM and SGLang work.

**So:** we can run the full model **today on our one 8× host**; we cannot run it broadly
without more 8× Blackwell capacity (us-central1 16-GPU quota; other-region 8× Spot was
partly stockout in the last bring-up).

## Recommendation

1. **Pilot now on the `g4-standard-384` 8× host:** deploy `nvidia/GLM-5.2-NVFP4` (TP-8, the
   card's exact flags, `transformers>=5.3.0`) and run the OpenCode-style tool loop + the
   #6310 reproduction. **Primary question: does the full model tool-call cleanly where the
   REAP checkpoint returns `provider_error`?** If yes, this is simultaneously the #6310 fix
   and the quality upgrade.
2. **If the pilot passes:** route Khala's GLM lane (at least the tool-bearing/coding traffic)
   to the full model on the 8× host; keep REAP-504B on the 4× hosts as overflow/plain-chat;
   pursue 8× Blackwell quota + capacity (#6311) to scale the full model.
3. **Track under #6316** (maximize GLM usage) and **#6310** (tool-calling). This evaluation
   is the input to the #6314-style precedence decision.

## Risks / caveats

- **8× host scarcity:** only one such host today; the full model can't run on the 4× majority
  — so without quota/capacity it's a single-replica lane (durability + throughput limited).
- **Bigger + heavier:** 753B vs 504B → more VRAM, lower per-host concurrency than the pruned
  model; the throughput-per-GPU tradeoff vs REAP must be measured (ties to #6312/#6320).
- **`transformers>=5.3.0` + `glm_moe_dsa`** must be in our serving image (a build change).
- NVIDIA tested on B200/B300 (180/192 GB); our RTX PRO 6000 (96 GB) is smaller per-GPU, so
  8× is the floor — validate KV/context headroom for our target context length.
- Standard NVFP4-vs-FP8 quality is near-parity per NVIDIA; still re-verify on our own
  OpenCode/Terminal-Bench traffic before making it primary.

### 1. Feasibility verdict — GO (capacity), single-host only

Re-confirmed against our recorded fleet facts (read-only):

- **Datatype:** RTX PRO 6000 Blackwell is **NVFP4-capable** and we already serve
  `modelopt_fp4` on it — no new hardware risk on the quant path.
- **VRAM fit:** full NVFP4 weights ≈ **381 GB**; shared expert is left **unquantized**
  (extra, non-trivial headroom), plus KV at our target context. The **`g4-standard-384`
  (8× RTX PRO 6000 = 768 GB)** host fits this at **TP-8**. The nine **`g4-standard-192`
  (4× = 384 GB)** hosts do **not** (weights ≈ all VRAM, no KV headroom) — they stay on
  REAP-504B. This matches the card's own `--tensor-parallel-size 8`.
- **Per-GPU caveat:** NVIDIA validated on B200/B300 (180/192 GB). Our RTX PRO 6000 is
  **96 GB/GPU**, so **TP-8 is the floor**, not a comfortable margin — the pilot must
  **measure the usable KV/context ceiling** (start conservative on `--max-model-len`,
  do **not** assume the model's 1M context fits at TP-8 on 96 GB cards).
- **Software:** the serving image needs **`transformers>=5.3.0`** (the `glm_moe_dsa`
  arch). This is an **image build change**, isolated to the pilot host's image — it does
  **not** require touching the live REAP fleet's running config.
- **Capacity bound:** exactly **one** 8× host today → the full model is a **single-replica
  lane** (durability + throughput ceiling limited). Broad rollout depends on **8× Blackwell
  quota/capacity (#6311)**; us-central1 RTX PRO 6000 Spot quota is **16 GPUs** (= two 8×
  hosts max in-region before a quota increase).

**Verdict: feasible today on the one 8× host; not feasible fleet-wide without #6311.**

### 2. Is it a credible #6310 fix + quality upgrade? — Yes (credible, must be proven)

- **#6310 is not a parser-config bug.** Our REAP launch already uses the **exact**
  `--tool-call-parser glm47 --reasoning-parser glm45` NVIDIA specifies, yet tool requests
  return `provider_error`. That isolates the fault to **the REAP-pruned checkpoint** (or a
  vLLM tool-path interaction specific to it), not our gateway config.
- **The full model is the controlled variable swap:** same parsers, same quant family
  (NVFP4), same engine — but the **full, agentic-validated** weights (τ²-Bench Telecom
  **98.25**; calibrated on **~992K tool-calling trajectories + Nemotron-SFT-SWE-v2**). If
  tool-calling is checkpoint-damage from REAP pruning, the full model should round-trip
  clean where REAP errors. This is **the** thing the pilot proves.
- **Quality upgrade:** NVFP4 is within noise of FP8 (better on 3/5 of NVIDIA's benches),
  and full ≫ pruned. Credible upgrade — but **re-verify on our own OpenCode / Terminal-Bench
  traffic**, not NVIDIA's benchmarks, before it becomes primary.

### 3. Concrete bounded pilot plan (owner / serving-lane executes; not me)

**Host:** the single `g4-standard-384` (8× RTX PRO 6000) in `us-central1`. **Isolate the
pilot** — bring it up as a **separate endpoint** (new port / new replica id), do **not**
repoint a live REAP replica and do **not** touch the gateway/`wrangler.jsonc` GLM lane.
Keep the live `openagents/khala` path on REAP-504B for the entire pilot.

**Image:** build/select a serving image with **`transformers>=5.3.0`** (the `glm_moe_dsa`
arch) + current vLLM (or SGLang). NVFP4 path is already exercised by the REAP image.

**Launch (vLLM, the card's exact flags + a measured context bound):**

```
vllm serve nvidia/GLM-5.2-NVFP4 \
  --tensor-parallel-size 8 \
  --quantization modelopt_fp4 \
  --tool-call-parser glm47 \
  --reasoning-parser glm45 \
  --trust-remote-code \
  --chunked-prefill-size 131072 \
  --mem-fraction-static 0.80 \
  --max-model-len <START 65536, raise only if KV headroom proven on 96GB cards> \
  --max-num-seqs 1   # start single-flight to isolate correctness; raise later for throughput
```

> Notes: `--mem-fraction-static 0.80` and `--chunked-prefill-size 131072` are the card's
> values. `--max-model-len` is **ours to bound** — the card's 1M context was validated on
> 180/192 GB cards; on 96 GB cards prove the KV ceiling empirically before raising it.
> SGLang equivalent: the card's SGLang recipe with `--tp 8` and the same parsers.

**Eval steps (in order):**

1. **Boot + load proof:** model loads at TP-8 within the host; record actual VRAM/GPU and
   the max `--max-model-len` that boots with usable KV headroom. If it OOMs at the chosen
   context, lower it — do **not** grow VRAM by touching other hosts.
2. **Tool-calling (PRIMARY test):** run the **OpenCode-style tool loop** (define
   `grep`/`glob`/`read`/`edit`, give a real task) **directly against this pilot endpoint**
   (bypassing the live gateway). Then replay the **#6310 reproduction** verbatim: the
   forced `get_weather` tool call that returns `{"error":"provider_error"}` on REAP, and a
   `tool_choice:auto` request. **Success = clean `tool_calls` round-trip, 0 `provider_error`,
   0 calls to nonexistent tools (no hallucinated `search`), over N≥20 consecutive requests.**
3. **Quality spot-check:** run a sample of our own **OpenCode / Terminal-Bench** style
   traffic against the pilot endpoint and eyeball parity vs REAP (coherence, instruction
   adherence, code edits). Decision-grade Terminal-Bench (#6253) is a later, gateway-routed
   run — this is the raw spot-check only.
4. **Throughput:** measure **tok/s single-flight** and **per-host concurrency** as
   `--max-num-seqs` rises; compare to REAP-504B's recorded **~47 tok/s single-flight**.
   Expect the full 753B to be **heavier (lower per-GPU throughput)** — **quantify the
   tradeoff** (ties to #6312/#6320). Record the concurrency at which TTFT/P90 degrade.
