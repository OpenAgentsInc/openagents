# Pylon Account Linking Process

This document explains what Pylon account linking now does, what it does not
do, and how the link stays fresh after the first command succeeds.

Account linking is optional. Local Pylon bring-up, Nexus connectivity, earning
work, wallet state, and local Codex auth remain useful without linking. Link a
Pylon only when the signed-in OpenAgents web account should see and manage that
node under `https://openagents.com/pylon`.

## What Linking Gives The Web Account

Linking gives `openagents.com` dashboard visibility for a Pylon identity:

- the Pylon public key and `npub`
- the human-readable node label
- the latest runtime state and diagnostic summary
- the ready local model, if any
- eligible product ids reported by the local Pylon
- web-safe capability snapshots, currently including `codex_agent`
- the active account-link row tying the observed Pylon to a signed-in user

Linking does not grant the hosted website raw access to the host machine.

The website cannot execute local code only because a Pylon is linked. Local
Codex auth stays on the Pylon host. Browser sessions, WorkOS cookies, ChatGPT
or Codex refresh tokens, local credential file paths, and raw local workspace
paths are not uploaded in the account-link or refresh payloads.

## The Two Machine-Facing Endpoints

The website owns two CSRF-free API routes for linked-node state.

### Initial Link Completion

`POST https://openagents.com/api/pylon-links/complete`

This endpoint is used by:

```bash
pylon account link --base-url https://openagents.com --token <one_time_token>
```

The request includes:

- the one-time token generated by the signed-in web user
- the Pylon public key and `npub`
- a runtime snapshot
- optional capability snapshots
- a NIP-98 `Authorization` proof signed by the node-held Pylon identity key

The website verifies both the token and the NIP-98 proof. The token proves the
signed-in user intentionally initiated a link. The NIP-98 proof proves the
request came from the node that owns the claimed public key.

If the token is valid, unexpired, and unused, the website creates or updates:

- `observed_pylons`
- `pylon_account_links`
- `pylon_capabilities`

If a different user already owns that Pylon, the active link is transferred
into historical state and the new account becomes the active owner.

### Linked Snapshot Refresh

`POST https://openagents.com/api/pylon-links/refresh`

This endpoint is used by:

```bash
pylon account refresh --base-url https://openagents.com
```

The refresh endpoint does not accept or consume a one-time token. It only works
for a Pylon identity that already has an active account link.

The request includes:

- the Pylon public key and `npub`
- the current runtime snapshot
- current capability snapshots
- the same NIP-98 proof shape signed by the node-held Pylon identity key

The website updates the observed Pylon row, upserts capability snapshots, and
refreshes `last_confirmed_at` on the active link. It refuses unknown or
unlinked Pylon identities so refresh cannot create ownership and cannot
transfer ownership.

`pylon codex workload once --base-url https://openagents.com --json` also
refreshes the linked-node snapshot before polling for a `pylon_codex`
assignment. That keeps Codex readiness fresh when the Pylon is actively
servicing web workloads.

## NIP-98 Proof

Both completion and refresh use NIP-98. The Pylon signs an event that binds:

- the exact absolute URL
- the HTTP method, always `POST` for link and refresh
- the SHA-256 hash of the raw JSON body bytes
- the Pylon public key

The website verifies:

1. The `Authorization` header decodes as a NIP-98 event.
2. The event id is correct.
3. The Schnorr signature verifies.
4. The event timestamp is fresh.
5. The `u` tag matches the exact request URL.
6. The `method` tag matches `POST`.
7. The `payload` tag matches the raw request body hash.
8. The event pubkey equals the JSON `public_key_hex`.

This prevents a copied body, copied token, or copied URL from being replayed by
someone who does not hold the Pylon identity key.

## Choosing The Right Binary

Use the standalone Pylon binary that owns the node identity.

These commands may resolve to package launchers:

- `npx @openagentsinc/pylon`
- `bunx @openagentsinc/pylon`
- `~/.bun/bin/pylon`

Launchers can start or update Pylon, but they may not expose every standalone
subcommand. A launcher error such as `Unknown argument: account` or
`--version requires a value` does not prove the running standalone Pylon cannot
link.

Prefer the same binary that is running or supervising the local node identity:

- installed standalone release binary
- cached standalone release binary
- source-built `target/debug/pylon`
- repo helper `cargo pylon-headless ...`

From a source checkout, the equivalent commands are:

```bash
cargo pylon-headless account link --base-url https://openagents.com --token <one_time_token>
cargo pylon-headless account refresh --base-url https://openagents.com --json
cargo pylon-headless codex workload once --base-url https://openagents.com --json
```

Before running a link command, confirm the selected binary is runnable:

```bash
/full/path/to/pylon --help
```

Do not reinstall, clone source, download model weights, or run Gemma
diagnostics just to link an account.

## Normal Operator Flow

1. Start or verify the local Pylon node.
2. Sign in to `https://openagents.com`.
3. Open `https://openagents.com/pylon`.
4. Generate a short-lived link command.
5. Run the generated `pylon account link ...` command with the standalone
   binary that owns the node identity.
6. Refresh `https://openagents.com/pylon`.
7. If the linked page later shows stale runtime or stale Codex capability,
   run `pylon account refresh --base-url https://openagents.com --json`.

If the Pylon is polling for web Codex assignments, the poll command performs
the refresh before claim:

```bash
pylon codex workload once --base-url https://openagents.com --json
```

The output should show either a claimed assignment or an idle report:

```json
{
  "claimed": false,
  "events_sent": 0,
  "completion_status": "idle"
}
```

Idle means no pending web workload was available for that linked identity. It
does not mean account linking failed.

## What The Web Page Shows

The linked-node page should distinguish these cases:

- `online`: the latest Pylon refresh reported an online runtime.
- runtime `error` or `degraded`: the local Pylon reported a current blocker.
- `Capability stale`: the last `codex_agent` capability snapshot is old.
- `Codex unavailable`: the linked Pylon has not reported a usable
  `codex_agent` capability.
- `needs_auth`: local Codex exists but needs local sign-in on the Pylon host.
- `ready`: local Codex is available for the conservative web workload lane.

The page should not treat an old snapshot as current truth. A successful
`account refresh` updates the timestamps and clears stale dashboard state when
the local Pylon is now healthy.

## Codex Capability Snapshot

Current Pylon builds report a `codex_agent` capability when they can inspect
the local Codex runner surface.

The web-safe capability fields include:

- `key`
- `status`
- `display_label`
- `auth_state`
- `runner_kind`
- `runner_version`
- `transport_kind`
- `supported_actions`
- `required_confirmations`
- `workspace_roots` hints, when safe
- non-secret metadata
- observed timestamps

The first web workload lane is conservative. It supports chat/read/patch
preview style work and keeps shell, file-write, and pull-request actions behind
explicit confirmation requirements for future execution policy.

## Troubleshooting

### Token Expired

Generate a fresh link command at `https://openagents.com/pylon`.

### Token Already Used

The one-time token was already consumed. If the first request reached the
server, the Pylon may already be linked. Run:

```bash
pylon account refresh --base-url https://openagents.com --json
```

Then refresh the web page.

### HTTP 419 / CSRF Token Mismatch

The completion or refresh request is hitting the wrong route stack or an old
deployment. The machine-facing endpoints must live under `/api` and return JSON
validation errors without a browser CSRF token.

Expected endpoints:

- `POST /api/pylon-links/complete`
- `POST /api/pylon-links/refresh`

### `Unknown argument: account`

The shell may be invoking a package launcher instead of the standalone binary.
Find the running or cached standalone binary and invoke it by full path, or put
its directory first in `PATH`.

### Dashboard Still Shows Stale Codex Capability

Run:

```bash
pylon account refresh --base-url https://openagents.com --json
```

If that succeeds, the web snapshot is fresh. If the page is still stale after
a reload, inspect the website deployment and API route table.

### Runtime Error Remains After Refresh

A refresh publishes current truth. If the page still shows an error, the local
Pylon is reporting a current runtime blocker. Fix the local blocker or accept
that the linked page is correctly showing degraded local state.

### Codex Is Ready But No Web Work Runs

Run:

```bash
pylon codex workload once --base-url https://openagents.com --json
```

If it returns `claimed: false`, there was no pending assignment for that
identity. If it fails before claim, read the account-refresh or workload claim
error. The poller refreshes account state before claiming work, so failures at
that stage are usually link, proof, or deployment issues.

## Product Boundary

The account-link flow is an account-management and dashboard-visibility flow.
It is not required for default local bring-up. It is not a wallet migration. It
is not Codex credential sync. It is not hosted remote code execution by itself.

The correct mental model is:

- local Pylon owns node identity, wallet state, runtime state, and local Codex
  auth
- `openagents.com` owns signed-in user identity and dashboard/account
  visibility
- NIP-98 proves that a web-bound snapshot came from the linked Pylon identity
- one-time tokens decide initial user ownership
- refresh keeps dashboard state current after ownership exists
