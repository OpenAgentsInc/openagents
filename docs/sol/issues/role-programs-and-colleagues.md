# P2 SARAH-ROLES: typed role programs and evidence-based colleague split

## Priority posture

The minimum decoded relationship/operator posture belongs in FC-1/FC-3 now.
This fuller lane begins after #8640 Phase A and uses canary evidence; it does not
block the coding cutover.

## Outcome

Sarah changes posture, retrieval, tools, and presentation from typed policy and
can delegate to a separately named persistent colleague only when doing so
clarifies a real responsibility or trust boundary.

## Scope

1. Generalize the FC `relationshipMode` seam into versioned role programs with
   explicit tool policy, retrieval/confidentiality scope, schedule,
   responsibility, success metric, escalation owner, and receipt grammar.
2. Keep role labels non-authoritative: compiled policy and the owning service
   decide what may happen.
3. Instrument repeated mode-switch tasks for user confusion, mistaken
   authority, accountability loss, and return-path preference.
4. Create a separately named colleague only when at least two of these
   materially diverge from Sarah's normal relationship and repeated tests show
   confusion or accountability loss:
   - authority/approval posture;
   - retrieval/confidentiality scope;
   - durable responsibility/schedule;
   - audience/communication channel;
   - success metric/escalation owner.
5. Show the active role/colleague and handoff in conversation, Blueprint, and
   receipts without multiplying state or execution authorities.
6. If the trigger is not met, record that result and keep the capability as a
   typed Sarah role rather than inventing a mascot.

## Exit

Canary evidence evaluates the split criteria for at least one standing
responsibility. Either Sarah runs it through a typed role program with correct
scope/posture, or a named colleague is created with a receipted handoff and
clearly distinct responsibility. In both cases authority remains in typed
services, cross-role data access is tested, and the user can identify who is
accountable.
