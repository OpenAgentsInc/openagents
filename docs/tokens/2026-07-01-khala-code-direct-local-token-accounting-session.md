# Khala Code direct-local token accounting session

Date: 2026-07-01

## Purpose

This note records the implementation and backfill work done to make Khala Code
Desktop account for direct/local Codex usage in the global `/stats` token
counter, and to make local message-level reconciliation possible afterward.

It follows the scope laid out in
`docs/tokens/2026-07-01-tokens-served-audit-khala-stats-codex-scope-expansion.md`:
direct/local Codex usage is not Khala API traffic, but when it is served through
our product client it still belongs in the global served-token count under the
`direct_local` channel.

## Problem observed

Khala Code Desktop was driving Codex app-server turns locally, but those turns
were not appearing in the global `/stats` counter. The initial failure mode was
not a provider usage problem: Codex was emitting exact token count updates in
the local rollout JSONL. The gap was that Khala Code Desktop was not reliably
persisting and mirroring those usage updates to OpenAgents Stats.

There was also no local audit file that could answer, turn by turn:

- which user message was submitted by Khala Code Desktop;
- which assistant messages were generated by Codex;
- which Codex thread/turn ids tie them together;
- which token usage event ids/idempotency keys should exist in the Stats
  ledger;
- how many tokens `/stats` should count for the turn.

## Implementation changes

Commit `e054095cad` introduced direct app-server token telemetry:

- Khala Code Desktop listens for `thread/tokenUsage/updated` notifications from
  Codex app-server.
- It records real token deltas immediately instead of waiting for a clean
  `turn/completed` path.
- It skips duplicate cumulative usage updates.
- It stores local usage JSONL rows at
  `~/.khala-code/token-usage-events.jsonl`.
- It mirrors canonical `openagents.token_usage_event.v1` events to
  `POST /api/stats/token-usage/events` when a bearer is configured.
- It records remote-post failures at
  `~/.khala-code/token-usage-report-failures.jsonl`.

Commit `d2059f5184` added message-token audit records:

- Every Khala Code Desktop Codex turn now writes a local audit row in `finally`,
  including interrupted turns and turns missing token updates.
- The audit row records the Khala client user message and Codex-generated
  messages with ids, roles, source labels, character counts, and SHA-256 hashes.
- The row records Codex thread/turn ids, desktop session/turn ids, exact token
  usage deltas, and the event ids/idempotency keys used for Stats mirroring.
- The row includes `globalCountedTokens`, defined as `inputTokens +
  outputTokens`, which is the number the public `/stats` projection counts for
  direct-local rows.
- The default local audit ledger path is
  `~/.khala-code/message-token-audit.jsonl`.

The same commit added a historical reconstruction script:

```sh
bun run --cwd clients/khala-code-desktop backfill:message-token-audit
```

The script walks Codex session JSONL files, selects sessions with
`originator: "khala_code_desktop"`, reconstructs Khala-submitted user messages,
Codex assistant messages, and exact token usage deltas, and writes local audit
rows in the same schema. It supports:

- `--sessions-root <path>`
- `--output <path>`
- `--dry-run`
- `--exclude-turn-id <codexTurnId>`
- repeated `--aggregate-backfill-event-id <eventId>`
- repeated `--aggregate-backfill-idempotency-key <key>`

## Follow-up reconciliation: Codex Desktop originator thread

After the top-right thread token meter was added, the Khala Code UI showed one
thread as fully pending even though the earlier manual aggregate backfill had
completed. The thread was:

- Codex thread id: `019f1e71-d86a-7da1-aafe-761ad6bb2a6a`
- Title: `The chat composer does not expand vertically with tall text...`
- Counted tokens: `10,249,823`
- Count basis: `10,219,044` input tokens + `30,779` output tokens

The cause was not the UI counter. The Codex rollout for that thread carried
exact `token_count` events, but its `session_meta.originator` was
`Codex Desktop`, not `khala_code_desktop`, so the original historical backfill
intentionally skipped it.

A narrow script override was added for manual reconciliation of known
Khala-owned threads whose upstream originator label does not say
`khala_code_desktop`:

```sh
bun run --cwd clients/khala-code-desktop backfill:message-token-audit -- \
  --sessions-root ~/.codex/sessions/2026/07/01 \
  --include-session-id 019f1e71-d86a-7da1-aafe-761ad6bb2a6a \
  --only-session-id 019f1e71-d86a-7da1-aafe-761ad6bb2a6a \
  --aggregate-backfill-event-id token_usage_event.khala_code_direct_local_thread_reconcile.20260701.019f1e71 \
  --aggregate-backfill-idempotency-key khala-code-desktop:direct-local-codex:thread-reconcile:2026-07-01:019f1e71-d86a-7da1-aafe-761ad6bb2a6a-v1
```

This wrote three local audit rows for the thread and tied them to the production
Stats aggregate event:

- event id:
  `token_usage_event.khala_code_direct_local_thread_reconcile.20260701.019f1e71`
- idempotency key:
  `khala-code-desktop:direct-local-codex:thread-reconcile:2026-07-01:019f1e71-d86a-7da1-aafe-761ad6bb2a6a-v1`

## Current invariant

Every future message sent through Khala Code Desktop must create two local
artifacts:

1. a token usage event row in `~/.khala-code/token-usage-events.jsonl`; and
2. a message-token audit row in `~/.khala-code/message-token-audit.jsonl`.

When Stats mirroring is configured, every nonzero token usage event must also
be mirrored to `POST /api/stats/token-usage/events` with an idempotency key.
The local audit row must contain the event ids/idempotency keys needed to
manually reconcile the local message history against the Stats database.

If a turn finishes without a token usage update, Khala Code Desktop still writes
a message-token audit row with `reconciliation.status:
"missing_token_usage_update"` so the missing count is explicit instead of
silent.
