## Working Title

Khala on Apple Silicon: Turn Every Mac into a Khala Node

## One-Line Hook

Every admitted Apple Silicon Mac has a local model sitting idle; Khala turns
that idle edge compute into a local node for chat, orchestration, continual
learning, training support, and eventually Bitcoin-paid useful work.

## Cold Open

Open on the question from Episode 194: what percentage of global AI workloads
will move from cloud inference to Apple Silicon? The answer is not only an
analyst question about GPU demand. It is a product question.

If Apple has put a capable on-device model into millions of machines, the
missing piece is not the chip. The missing piece is a network: discovery,
identity, routing, receipts, trust, and a user interface that lets a normal Mac
owner participate without becoming an infrastructure operator.

That is what Khala Desktop is for.

## Thesis

Khala should make every admitted Apple Silicon Mac a self-contained Khala node:
a local chat app, a local Pylon, a local Apple Foundation Models backend,
OpenAgents identity, truthful fleet status, and an optional provider mode that
can advertise only the capabilities the machine actually has.

The old cloud-only pattern is simple: a user sends tokens to a datacenter and
pays the API bill. The Khala pattern is different: the user already owns a
machine, the machine already has spare cycles, and the network can route useful
micro-work to it when it is ready. That makes the Mac part of the collective
intelligence fabric instead of only a terminal pointed at someone else's cloud.

This is the Apple Silicon version of the larger Khala thesis from Episode 244:
one orchestration layer behind one API, able to route across the user's own
resources first and across the wider network later.

## Segment 1: The Trillion-Dollar Question Was a Routing Question

Episode 194 framed the macro question: if Apple Silicon can run a meaningful
share of AI inference by 2030, the economics of everyday AI change. The same
episode named developer adoption of Apple Foundation Models plus MLX as one of
the signals to watch.

The important follow-up is practical. A local model does not become useful at
global scale merely because it exists. It needs an app surface developers and
normal users will run. It needs a way to prove availability. It needs receipts
and routing so work can find it. It needs honest unsupported states so a Mac
that cannot run Apple FM is not treated as broken or ready.

Khala on Apple Silicon is the product answer to that question:

- local first, because the machine is already there;
- networked, because isolated local models do not create a market;
- receipt-backed, because useful work needs proof;
- honest about readiness, because unsupported machines are normal.

## Segment 2: Fracking Apple Silicon

Episode 201 gave the metaphor: Apple Silicon is stranded compute. The resource
exists, but it needs a market path. In oil language, fracking is not the oil; it
is the process that makes a hard-to-reach resource flow.

For AI, the process is:

1. Discover the local machine and its capabilities.
2. Start or attach to a local Pylon.
3. Check Apple Foundation Models availability through the bridge.
4. Publish truthful capability refs only when the machine is admitted.
5. Route bounded work to the node.
6. Record receipts, usage truth, closeouts, and proofs.

That is why the Khala Desktop spec is not just "make a Mac chat app." A chat
window is table stakes. The real product is the node lifecycle: app opens,
Pylon boots, Apple FM readiness is checked, the user can chat immediately, and
provider mode only goes online when the local facts support it.

## Segment 3: What the macOS App Boots

The proposed Khala Desktop app is native SwiftUI, not a web wrapper and not
Expo. It should feel like the Mac sibling of the current native Khala iOS app,
but with more room for local node operations.

The first screen should be useful even before provider mode is perfect:

- a Khala chat surface backed by the public `openagents/khala` route;
- local conversation history and Keychain-backed auth;
- visible local Pylon readiness;
- Apple FM status: ready, unavailable, unsupported, or owner-action-required;
- fleet/provider status for connected accounts and local capacity;
- a go-online control that publishes only verified capability refs.

The app should support two Pylon modes. If a suitable local Pylon already runs,
attach to it. If not, boot an app-managed Pylon home under Application Support.
That Pylon home must not touch unrelated local credentials or default Codex
homes. Local diagnostics can exist, but public views must stay redacted.

For Apple FM, the app should reuse the existing bridge contract instead of
inventing a new local-model path: a loopback bridge, a health check, typed
availability, the `apple_fm_bridge` family, the `apple-foundation-model` model
id, and usage truth that can be exact, estimated, or unknown. Never fake exact
tokens.

## Segment 4: Why This Matters for Continual Learning and Tassadar

Khala is not only a cheaper chat endpoint. It is a way to coordinate many
small pieces of useful work. That matters for continual learning, training
data, replay, validation, and Tassadar-style exact-execution workflows.

The Apple Silicon node is a good fit for small local jobs:

- summarizing and distilling local-safe traces;
- running bounded tool or classification tasks;
- assisting local orchestration before cloud escalation;
- contributing micro-results to larger training or evaluation loops;
- staying available overnight as idle capacity.

The claim is not that Apple FM replaces Codex, Claude Code, frontier models, or
training clusters. The claim is more important and easier to prove: a large
amount of useful agent work is small, parallel, local, or orchestration-heavy.
If Khala can route that work to machines people already own, the network gets
more useful without waiting for another datacenter.

For Tassadar, that means the run can eventually see not only cloud providers
and paid coding agents, but also local edge contributors: Macs that can help
with safe fragments of the larger verification and learning loop, with proofs
and receipts deciding what counts.

## Segment 5: The Honest Button

Episode 214 had the simple user promise: install the software, click "Go
Online," and sell spare compute for Bitcoin. That remains the north star, but
the honest implementation has to be stricter than the slogan.

The button should not mean "pretend my Mac can do anything." It should mean:

- my Pylon is online;
- my identity is linked;
- my capability refs are fresh;
- Apple FM is ready only if the bridge and environment say it is ready;
- assignments have closeouts and proofs;
- earnings distinguish unsettled, rejected, credited, and settled states.

That honesty is what makes the network durable. Unsupported Apple FM is not a
failure; it is a truthful state. Unknown usage is not exact usage. A local
estimate is not a public counter row. This is how Khala can scale without
turning into vibes.

## Segment 6: Call to Action

The ask for Mac owners is simple: when the Khala Desktop path is ready, run it.
Bring an admitted Apple Silicon machine online. Let it become one more node in
the collective intelligence.

The ask for developers is sharper:

- help build the native SwiftUI desktop app;
- reuse the existing Pylon and Apple FM bridge contracts;
- improve fake-bridge tests and admitted-Mac smoke coverage;
- keep provider claims receipt-backed and public-safe;
- find the smallest useful local jobs that Apple Silicon can handle reliably.

The ask for the network is bigger: stop treating user-owned hardware as only a
client for cloud APIs. A Mac can be a node. A node can serve useful work. Useful
work can produce receipts. Receipts can become reputation, routing, training
signal, and eventually payment.

## Closing

Episode 194 asked the trillion-dollar question. Episode 201 named the strategy:
frack Apple Silicon. Episode 244 showed Khala routing coding work through the
owner's own capacity.

Episode 246 should make the next step concrete: Khala Desktop is the app that
boots the local node. It is the path from "my Mac has an idle model" to "my Mac
is part of Khala."

The future is not only bigger clouds. It is millions of small machines doing
bounded useful work, telling the truth about what they did, and getting paid
when that work matters.
