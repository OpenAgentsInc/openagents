### Verified Buzz and Nostr sources

Block's public [`block/buzz`](https://github.com/block/buzz) repository is the
primary Buzz source for this episode. The research snapshot uses
[`3d17195`](https://github.com/block/buzz/tree/3d17195ac88a3fd0c2fb29d6208e35d0905050bc).
Its README identifies these capabilities as working today:

- relay, channels, threads, direct messages, canvases, media, search, and an
  audit log
- a Tauri and React desktop application
- `buzz-cli` and an ACP harness for Goose, Codex, and Claude Code
- workflows triggered by messages, reactions, schedules, and webhooks
- NIP-34 Git events and a Git hosting backend

The same README marks mobile clients and complete workflow approval gates as
unfinished. It describes a feature branch as a room where patches, CI, review,
conversation, and the merge decision stay together. It also says the relay is
the single source of truth. Do not describe Buzz as a peer-to-peer company
database or claim that all of its roadmap is shipped.

Buzz makes retained channel history queryable, but a newly added agent does not
automatically ingest that history. Its first live subscription starts at the
present time. The agent must fetch recent messages or search. Do not repeat
`instantly reads everything` as a product fact.

Buzz gives an agent a separate keypair and separate channel membership. Its
virtual-agent design does not inherit the owner's membership, role, or
administrator rights. This is a useful boundary for Omega. Also, Buzz private
channels use relay-enforced read access. Do not imply that every workroom event
is end-to-end encrypted from the relay operator.

Nostr references:

- [NIP-01](https://github.com/nostr-protocol/nips/blob/master/01.md) defines
  events, public keys, event identifiers, Schnorr signatures, and relay
  subscriptions.
- [NIP-05](https://github.com/nostr-protocol/nips/blob/master/05.md#identification-not-verification)
  says that its identifiers provide identification, not verification.
- [NIP-29](https://github.com/nostr-protocol/nips/blob/master/29.md) defines
  relay-based groups and group membership events.
- [NIP-34](https://github.com/nostr-protocol/nips/blob/master/34.md) defines Git
  repository, patch, and status events.
- [NIP-42](https://github.com/nostr-protocol/nips/blob/master/42.md) defines
  relay authentication.
- [NIP-44](https://github.com/nostr-protocol/nips/blob/master/44.md) defines an
  encrypted payload format and documents important limits, including no
  forward secrecy and metadata leakage.
- [NIP-46](https://github.com/nostr-protocol/nips/blob/master/46.md) defines a
  remote-signer pattern that can keep an identity key outside a client.

NIP-01 signatures show that a key signed exact event bytes. They do not grant
application authority, approve a deployment, authorize payment, prove that a
claim is true, or make public content private. The edit must preserve that
distinction.
