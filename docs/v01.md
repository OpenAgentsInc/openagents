# v0.1 Release Cut

Date: Tuesday, March 10, 2026  
Target release: Wednesday, March 11, 2026  
Status: Release target for tomorrow

This document defines the production cut for `v0.1`.

`v0.1` lets anyone with an Apple Silicon Mac click `Go Online` and sell local
compute for bitcoin.

For `v0.1`, we are intentionally shipping a narrower slice than the broader MVP
docs describe. The goal is not to preserve the full pane workspace. The goal
is to bootstrap the compute market with the smallest honest loop that can work
tomorrow.

That loop is:

1. Open the desktop app.
2. Land directly in Mission Control.
3. Bring Apple Foundation Models ready if needed.
4. Press `Go Online`.
5. Receive a paid starter/open-market compute job.
6. See wallet-confirmed sats increase.
7. Withdraw over Lightning.

If a feature does not help that loop happen on March 11, 2026, it is out of the `v0.1` release cut.

Just as important: `v0.1` is not intended to be a real buyer product for
serious work. The buyer-facing surfaces that exist in the app are there to
bootstrap and validate the seller network:

- prove jobs can be assigned to a specific online seller
- prove the seller can execute and publish a result
- prove a buyer wallet can pay
- prove the seller is actually paid and the wallet/accounting truth updates

If a buyer-facing feature does not help validate that seller-market loop, it is
not part of the `v0.1` release purpose.

## Authority And Overrides

`docs/MVP.md` remains the product authority for the retained MVP.

For the `v0.1` release cut, this document overrides broader docs anywhere they imply:

- a production multi-pane workspace
- a production hotbar/grid shell
- GPT-OSS as part of the release UX
- a need to leave Mission Control to complete the first earn loop

In particular, `v0.1` narrows the current expectations in:

- `docs/UI.md`
- `docs/PANES.md`
- `docs/plans/mission-control-pane.md`

Those docs can remain broader for ongoing development, but the shipped default for tomorrow is narrower.

## Product Decision

`v0.1` is `Mission Control` only.

The production app should open into one fullscreen Mission Control surface and nothing else. No pane maze. No grid workspace. No floating-pane desktop metaphor. No hotbar as part of the release UX.

This is a release about bootstrapping supply for the compute market, not about showcasing the general pane system.

It is also a release about bootstrapping the seller network, not about shipping
a polished buyer workstation. The buyer path in `v0.1` exists to create,
target, and settle real jobs so provider assignment and payment are proven
under live conditions.

## No GPT-OSS, Just Apple FM

The release runtime story is:

- Apple Silicon Mac
- Apple Foundation Models
- text-generation-led compute jobs
- Mission Control as the only production shell

For `v0.1`:

- no GPT-OSS release path
- no GPT-OSS copy in the production Mission Control surface
- no GPT-OSS workbench in the production navigation story
- no "use the separate GPT-OSS pane" fallback in the release UX

If Apple FM is unavailable, Mission Control should say so plainly and truthfully. It should not redirect the user into a second runtime story for the release.

Non-macOS and alternate runtime lanes may continue to exist for development, but they are not part of the `v0.1` release promise.

## One-Screen Mission Control

The current Mission Control content is already the right nucleus. The shell is what must change.

For `v0.1`, Mission Control stops behaving like one pane inside a larger workspace and becomes the app itself.

Production Mission Control should own all of the first-session earn loop inline:

- Apple FM readiness and start/refresh action
- provider mode state and `Go Online`
- wallet connection and balance truth
- market activity / starter-job preview while offline
- active job visibility
- earnings visibility
- withdrawal entry point
- optional inline `Buy Mode` smoke-test controls when explicitly enabled
- plain-language blockers and recovery hints

The user should not need to open:

- a separate Apple FM workbench
- a wallet pane
- an identity pane
- a settings pane
- a command palette
- any other pane

for the core `v0.1` loop.

## Buy Mode Smoke-Test Lane

`v0.1` remains earn-first. `Buy Mode` is not a second product shell, not a
generic buyer console, and not a reason to widen the release UX back into a
multi-pane workspace.

It is a small Mission Control smoke-test lane for staging, QA, bilateral
verification, and operator debugging of the real marketplace path. When
armed, it should keep publishing tiny paid requests on a fixed cadence and
watch the buyer flow settle end to end without leaving Mission Control.

The important product boundary is this: `Buy Mode` is not here to make the app
useful as a general-purpose buyer client. It is here to verify the seller
network. A good `v0.1` buyer flow is one that reliably proves:

- a seller was selected
- the job was assigned
- the seller produced a result
- the buyer paid
- the seller was actually paid
- the UI and logs tell the truth about each of those steps

Contract:

- visible by default in Mission Control
- primary action: `START BUY MODE` / `STOP BUY MODE`
- publishes one `kind: 5050` NIP-90 request using a fixed tiny prompt template
  chosen for cheap validation
- fixed spend per request: `2 sats`
- current code default dispatch cadence: `100ms`
- one in-flight buy-mode request at a time
- current code targets one eligible Autopilot peer pubkey at a time instead of
  broadcasting the request to every reachable provider
- inline lifecycle: `published -> feedback -> result -> invoice paid ->
  settled/failed`
- uses the built-in Spark wallet as the buyer wallet
- reuses the existing app-owned network-request state rather than inventing a
  second buyer state model

This lane complements hosted starter demand. It does not replace the
OpenAgents-run starter buyer loop that guarantees first earnings for providers.
Its job is to prove that the desktop can also originate a real tiny paid
request on demand.

## UI Shell Rule

The current grid + hotbar desktop shell moves behind a dev-only gate.

Production default:

- fullscreen Mission Control
- no dots-grid workspace chrome
- no hotbar
- no floating pane frame for Mission Control
- no pane dragging/resizing requirement
- no alternate pane navigation in the normal release flow
- the buy-mode controls render inline inside Mission Control rather than
  opening a second buyer pane

Dev mode only:

- existing pane workspace
- hotbar
- grid background
- command palette driven pane hopping
- separate workbench/debug panes

Recommended gate shape: a single app-level env flag following the existing feature-gate pattern, for example `OPENAGENTS_ENABLE_DEV_MODE=1`.

Default must be off.

## Minimal Implementation Direction

This should stay app-owned in `apps/autopilot-desktop`, consistent with `docs/OWNERSHIP.md`.

Minimal implementation path:

1. Extend pane metadata in `apps/autopilot-desktop/src/pane_registry.rs` with a presentation flag, preferably something as small as `fullscreen: bool` or an equivalent `PanePresentation` enum.
2. Mark `PaneKind::GoOnline` / Mission Control as fullscreen in the production path.
3. Teach the app-owned pane orchestration/render path to honor that presentation mode without moving product logic into `crates/wgpui`.
4. Gate the current workspace shell in `apps/autopilot-desktop/src/render.rs`, `apps/autopilot-desktop/src/hotbar.rs`, and related input handling so it only appears in dev mode.
5. Keep separate panes available only for internal testing and development.
6. Reuse `NetworkRequestsState`, `NetworkRequestsPaneInputs`, and the existing
   Spark wallet send lane to render a small inline buy-mode block inside
   Mission Control, with env override only if we explicitly want to suppress it.
7. Keep the buy-mode contract fixed to a one-shot `kind: 5050` request at
   `2 sats`; do not turn the release cut into a generic buyer composer.

Do not turn this into a large UI rewrite. Reuse the existing Mission Control sections and state sources. Change the shell, not the truth model.

## Current Repo Grounding

This release cut is grounded in the current retained implementation:

- `apps/autopilot-desktop/src/pane_registry.rs` makes Mission Control (`PaneKind::GoOnline`) the singleton startup pane.
- `apps/autopilot-desktop/src/pane_system.rs` and `apps/autopilot-desktop/src/pane_renderer.rs` now treat Mission Control as the fullscreen production shell.
- `apps/autopilot-desktop/src/render.rs`, `apps/autopilot-desktop/src/input.rs`, and `apps/autopilot-desktop/src/desktop_shell.rs` gate the dots-grid, hotbar, and command palette behind `OPENAGENTS_ENABLE_DEV_MODE`.
- `apps/autopilot-desktop/src/app_state.rs` and `apps/autopilot-desktop/src/input/actions.rs` now keep Mission Control on Apple FM truth in production and keep the inline withdraw path inside the same screen.
- `apps/autopilot-desktop/src/runtime_log.rs` and `apps/autopilot-desktop/src/desktop_control.rs` now persist structured session logs, `latest.jsonl`, and a `desktop-control.json` manifest for UI-synced automation and audit.
- `apps/autopilot-desktop/src/nip28_chat_lane.rs` and `apps/autopilot-desktop/src/autopilot_peer_roster.rs` now maintain a managed NIP-28 main-channel lane plus an Autopilot compute-presence roster.
- `apps/autopilot-desktop/src/input/actions.rs` now builds targeted buy-mode requests from that NIP-28 roster instead of emitting untargeted broadcast requests.
- `apps/autopilot-desktop/src/bin/autopilotctl.rs` and `apps/autopilot-desktop/src/desktop_control.rs` now provide a synced CLI/control-plane path that drives the same Mission Control truth model as the UI.
- That synced CLI/control-plane path is now lane-aware for local-runtime automation: the packaged release checks still drive Apple FM, while generic `local-runtime` and GPT-OSS-specific status/refresh/warm/unload verbs exist for retained NVIDIA hosts.
- `scripts/release/check-v01-packaged-autopilotctl-roundtrip.sh` now validates the bundled app and a runtime app together through chat, buyer flow, seller flow, and Spark settlement.
- Separate panes and alternate runtime/debug paths still exist for internal and dev-mode use.

`v0.1` should preserve the existing Mission Control data truth while collapsing the shell around it.

## Actual Codebase Status (March 12, 2026)

This section is the honest release snapshot of what is actually present in
`main` now. It is descriptive, not aspirational.

### What now matches the original v0.1 cut

- The production default shell is still Mission Control-first. The app lands in
  the Mission Control singleton/startup pane, and the production shell still
  suppresses the old workspace chrome unless `OPENAGENTS_ENABLE_DEV_MODE=1`.
- Apple Foundation Models remains the production local-runtime truth. GPT-OSS
  support still exists in code, but only as a dev-mode/internal path.
- Wallet truth, withdrawal, provider state, active-job state, market preview,
  and buy-mode state are all still projected into Mission Control instead of
  requiring another pane to complete the core earn loop.
- The seller-side lifecycle is now fully explicit in the app state, UI, and
  runtime logs: receive -> accept -> run -> deliver -> request payment ->
  settle -> paid/failed.
- The release is still seller-first in spirit: the strongest parts of the app
  are the provider lifecycle, payout truth, payment settlement, and operator
  verification around those paths.

### What changed relative to the original v0.1 plan

- `Buy Mode` is now visible by default. `OPENAGENTS_ENABLE_BUY_MODE` still
  exists, but current code treats it as an override to hide buy mode rather
  than a gate required to reveal it.
- The original `12s` buy-mode cadence in this doc is no longer true. Current
  code defaults to a `100ms` cadence. That change came from throughput work and
  materially changed the product behavior.
- `Buy Mode` no longer blindly throws one NIP-90 request at every connected
  provider. Current code selects one eligible Autopilot peer from the managed
  NIP-28 main-channel roster and targets that peer's pubkey.
- NIP-28 is now part of the real operational story even though the original
  v0.1 plan did not make it central. The desktop maintains a managed main
  channel, emits/consumes structured compute presence
  (`oa.autopilot.presence.v1`), tracks an Autopilot peer roster, and uses that
  roster to decide where buy-mode jobs go.
- The app now has an app-owned control plane. `autopilotctl` can steer the
  running UI through `desktop_control.rs`, read snapshots, wait on conditions,
  read event batches, and drive Mission Control programmatically.
- File-backed observability is now part of the actual product/operator surface.
  Session logs, `latest.jsonl`, and `desktop-control.json` are written by
  default and are used by both verification and debugging flows.
- The buyer side is still not a polished end-user work surface. In practice,
  the buyer flow has evolved into an operator/bootstrap lane whose main job is
  to prove seller selection, assignment, settlement, and payout truth.

### What we now have in the app beyond the original v0.1 promise

- A full NIP-28 managed-chat runtime in
  `apps/autopilot-desktop/src/nip28_chat_lane.rs`.
- An Autopilot peer-roster/presence system in
  `apps/autopilot-desktop/src/autopilot_peer_roster.rs`.
- A UI-synced desktop control runtime in
  `apps/autopilot-desktop/src/desktop_control.rs`.
- A first-class CLI client for that control runtime in
  `apps/autopilot-desktop/src/bin/autopilotctl.rs`.
- A packaged-app roundtrip harness in
  `scripts/release/check-v01-packaged-autopilotctl-roundtrip.sh` that drives a
  bundled app and a runtime app through:
  - Apple FM readiness
  - provider online/offline
  - NIP-28 main-channel selection
  - bidirectional chat
  - targeted buy-mode dispatch
  - Spark-funded buyer and seller payment settlement
- A throughput benchmark harness in
  `apps/autopilot-desktop/src/throughput_bench.rs` and
  `apps/autopilot-desktop/src/bin/autopilot_throughput_bench.rs`.
- Headless compute and Spark CLI tooling that remain useful for operator and
  integration verification, even though they are not the end-user GUI story.

### What is still broader than the shipped end-user release story

- The repo still contains the wider pane system, workbenches, debug panes, and
  internal shells. They are no longer the default product shell, but they still
  exist in the codebase and can be surfaced in dev mode.
- The codebase is now broader than a pure one-screen retail product. It also
  contains operator/agent surfaces: `autopilotctl`, desktop-control HTTP
  control, headless compute flows, throughput tools, and durable JSONL logs.
- NIP-28 support is operationally real now, but it should be described
  honestly in release notes: today it is primarily an Autopilot coordination
  and targeting layer, not a polished consumer chat product.

### Release honesty check

If we release from the current codebase, the honest statement is:

- the default GUI experience is still Mission Control-first and Apple FM-first,
- the seller earn loop is real,
- the buyer smoke-test lane is real, but it is there to validate seller
  assignment and payment rather than to serve as a serious buyer product,
- programmatic control of the running UI is real,
- durable file-backed logs are real,
- and NIP-28 is now a meaningful part of the compute-market path because it is
  how Autopilot peers announce availability and how buy-mode targets a specific
  online peer.

That is stronger than the original v0.1 plan, but it is also different from
what this document originally described. Release notes and operator docs should
say that plainly.

## Scope For Tomorrow

`v0.1` should ship only what is required to make the Apple FM compute-market bootstrap loop work.

In scope:

- Apple FM-first Mission Control
- one-screen fullscreen shell
- inline runtime readiness
- inline wallet truth
- inline starter/open-market job visibility
- wallet-confirmed earning visibility
- withdraw path
- Mission Control `Buy Mode` loop for `kind: 5050` at `2 sats`

Out of scope for the release UX:

- GPT-OSS
- the general pane workspace as the default experience
- the hotbar as a production control surface
- "open another pane to continue" flows
- broad multi-runtime messaging
- a generic buyer request composer or RFQ workflow in the main release shell
- product polish work that does not improve the first paid-job loop

## Release Acceptance Criteria

`v0.1` is ready when all of the following are true:

- On a supported Mac, app launch lands in fullscreen Mission Control by default.
- The default release surface does not show the grid workspace or hotbar.
- The default release surface does not require opening any second pane to complete the first earn loop.
- Mission Control speaks only Apple FM truth in the production path.
- `Go Online` is gated by honest Apple FM, wallet, and provider readiness.
- Offline Mission Control still shows live or recent market activity.
- The user can receive at least one paid job and see wallet-confirmed sats tick up without leaving the screen.
- Mission Control Buy Mode can be toggled on and show inline publish,
  result, and payment outcome for repeated `kind: 5050` / `2 sats` jobs.
- Dev mode can still expose the old multi-pane shell for internal use.

## Release Verification

Run the explicit regression gate:

- `./scripts/release/check-v01-mission-control.sh`

Then perform the manual acceptance pass on a supported Mac:

1. Launch the app without `OPENAGENTS_ENABLE_DEV_MODE`.
2. Confirm the app lands in fullscreen Mission Control with no grid workspace, hotbar, or command palette.
3. Confirm the Mission Control local-model action only starts or refreshes Apple FM and does not require opening a second pane.
4. Paste a Lightning invoice into the Mission Control withdraw field and confirm the withdraw action is available on the same screen.
5. Go online, complete at least one paid job, and confirm wallet-authoritative sats increase without leaving Mission Control.
6. Relaunch with `OPENAGENTS_ENABLE_DEV_MODE=1` and confirm the old pane workspace, hotbar, and debug surfaces return for internal use.
7. Toggle `START BUY MODE`, confirm Mission Control starts dispatching `kind:
   5050` / `2 sats` jobs on cadence, and confirm the request publish, result,
   and buyer payment outcome stay inline.
