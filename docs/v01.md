# v0.1 Release Cut

Date: Tuesday, March 10, 2026  
Target release: Wednesday, March 11, 2026  
Status: Release target for tomorrow

This document defines the production cut for `v0.1`.

For `v0.1`, we are intentionally shipping a narrower slice than the broader MVP docs describe. The goal is not to preserve the full pane workspace. The goal is to bootstrap the compute market with the smallest honest loop that can work tomorrow.

That loop is:

1. Open the desktop app.
2. Land directly in Mission Control.
3. Bring Apple Foundation Models ready if needed.
4. Press `Go Online`.
5. Receive a paid starter/open-market compute job.
6. See wallet-confirmed sats increase.
7. Withdraw over Lightning.

If a feature does not help that loop happen on March 12, 2026, it is out of the `v0.1` release cut.

## Authority And Overrides

`docs/MVP.md` remains the product authority for the retained MVP.

For the `v0.1` release cut, this document overrides broader docs anywhere they imply:

- a production multi-pane workspace
- a production hotbar/grid shell
- GPT-OSS as part of the release UX
- a need to leave Mission Control to complete the first earn loop

In particular, `v0.1` narrows the current expectations in:

- `docs/UI.md`
- `docs/PANES.md`
- `docs/plans/mission-control-pane.md`

Those docs can remain broader for ongoing development, but the shipped default for tomorrow is narrower.

## Product Decision

`v0.1` is `Mission Control` only.

The production app should open into one fullscreen Mission Control surface and nothing else. No pane maze. No grid workspace. No floating-pane desktop metaphor. No hotbar as part of the release UX.

This is a release about bootstrapping supply for the compute market, not about showcasing the general pane system.

## No GPT-OSS, Just Apple FM

The release runtime story is:

- Apple Silicon Mac
- Apple Foundation Models
- text-generation-led compute jobs
- Mission Control as the only production shell

For `v0.1`:

- no GPT-OSS release path
- no GPT-OSS copy in the production Mission Control surface
- no GPT-OSS workbench in the production navigation story
- no "use the separate GPT-OSS pane" fallback in the release UX

If Apple FM is unavailable, Mission Control should say so plainly and truthfully. It should not redirect the user into a second runtime story for the release.

Non-macOS and alternate runtime lanes may continue to exist for development, but they are not part of the `v0.1` release promise.

## One-Screen Mission Control

The current Mission Control content is already the right nucleus. The shell is what must change.

For `v0.1`, Mission Control stops behaving like one pane inside a larger workspace and becomes the app itself.

Production Mission Control should own all of the first-session earn loop inline:

- Apple FM readiness and start/refresh action
- provider mode state and `Go Online`
- wallet connection and balance truth
- market activity / starter-job preview while offline
- active job visibility
- earnings visibility
- withdrawal entry point
- plain-language blockers and recovery hints

The user should not need to open:

- a separate Apple FM workbench
- a wallet pane
- an identity pane
- a settings pane
- a command palette
- any other pane

for the core `v0.1` loop.

## UI Shell Rule

The current grid + hotbar desktop shell moves behind a dev-only gate.

Production default:

- fullscreen Mission Control
- no dots-grid workspace chrome
- no hotbar
- no floating pane frame for Mission Control
- no pane dragging/resizing requirement
- no alternate pane navigation in the normal release flow

Dev mode only:

- existing pane workspace
- hotbar
- grid background
- command palette driven pane hopping
- separate workbench/debug panes

Recommended gate shape: a single app-level env flag following the existing feature-gate pattern, for example `OPENAGENTS_ENABLE_DEV_MODE=1`.

Default must be off.

## Minimal Implementation Direction

This should stay app-owned in `apps/autopilot-desktop`, consistent with `docs/OWNERSHIP.md`.

Minimal implementation path:

1. Extend pane metadata in `apps/autopilot-desktop/src/pane_registry.rs` with a presentation flag, preferably something as small as `fullscreen: bool` or an equivalent `PanePresentation` enum.
2. Mark `PaneKind::GoOnline` / Mission Control as fullscreen in the production path.
3. Teach the app-owned pane orchestration/render path to honor that presentation mode without moving product logic into `crates/wgpui`.
4. Gate the current workspace shell in `apps/autopilot-desktop/src/render.rs`, `apps/autopilot-desktop/src/hotbar.rs`, and related input handling so it only appears in dev mode.
5. Keep separate panes available only for internal testing and development.

Do not turn this into a large UI rewrite. Reuse the existing Mission Control sections and state sources. Change the shell, not the truth model.

## Current Repo Grounding

This release cut is grounded in the current retained implementation:

- `apps/autopilot-desktop/src/pane_registry.rs` already makes Mission Control (`PaneKind::GoOnline`) a singleton startup pane.
- `apps/autopilot-desktop/src/pane_system.rs` already has a concrete Mission Control layout.
- `apps/autopilot-desktop/src/app_state.rs` and `apps/autopilot-desktop/src/pane_renderer.rs` already treat Mission Control as FM-first on macOS.
- `apps/autopilot-desktop/src/render.rs` still paints the global dots-grid shell and hotbar around the pane system.
- `apps/autopilot-desktop/src/hotbar.rs` and `apps/autopilot-desktop/src/input.rs` still wire the hotbar and pane shortcuts globally.
- `apps/autopilot-desktop/src/input/actions.rs` still assumes Mission Control can open separate workbench panes.

`v0.1` should preserve the existing Mission Control data truth while collapsing the shell around it.

## Scope For Tomorrow

`v0.1` should ship only what is required to make the Apple FM compute-market bootstrap loop work.

In scope:

- Apple FM-first Mission Control
- one-screen fullscreen shell
- inline runtime readiness
- inline wallet truth
- inline starter/open-market job visibility
- wallet-confirmed earning visibility
- withdraw path

Out of scope for the release UX:

- GPT-OSS
- the general pane workspace as the default experience
- the hotbar as a production control surface
- "open another pane to continue" flows
- broad multi-runtime messaging
- product polish work that does not improve the first paid-job loop

## Release Acceptance Criteria

`v0.1` is ready when all of the following are true:

- On a supported Mac, app launch lands in fullscreen Mission Control by default.
- The default release surface does not show the grid workspace or hotbar.
- The default release surface does not require opening any second pane to complete the first earn loop.
- Mission Control speaks only Apple FM truth in the production path.
- `Go Online` is gated by honest Apple FM, wallet, and provider readiness.
- Offline Mission Control still shows live or recent market activity.
- The user can receive at least one paid job and see wallet-confirmed sats tick up without leaving the screen.
- Dev mode can still expose the old multi-pane shell for internal use.
