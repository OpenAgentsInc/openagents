# v0.2 Release Notes

Date: Wednesday, March 18, 2026  
Status: Upcoming release target  
Release theme: Data Market MVP launch

This document explains the current `v0.2` launch we are about to ship.

`v0.1` was the compute-market bootstrap release: go online, receive paid
compute work, watch wallet-confirmed sats land.

`v0.2` adds the first real Data Market lane.

That means OpenAgents can now do more than sell machine time. It can also sell
permissioned access to packaged local data through the same product family:

- a visible desktop UI
- a synced CLI control plane
- a no-window runtime for scripts and operators
- first-party agent skills for both in-app and shell-first flows

This is still a narrow MVP. It is a real launch, not a full public data
marketplace.

## What v0.2 Is

`v0.2` is the launch of a seller-first Data Market MVP.

You can now:

- package a local file or folder boundary into a truthful listing draft
- publish a `DataAsset`
- publish a targeted `AccessGrant`
- accept a targeted buyer request over NIP-90
- request payment when a priced grant requires it
- issue a `DeliveryBundle`
- let a buyer consume the delivered payload locally
- revoke or expire access and record a `RevocationReceipt`

The important product boundary is this:

- this is a permissioned context-vending lane
- it is targeted by default
- it is authority-backed
- it is not yet a broad searchable public marketplace

Compute remains the primary onboarding and monetization loop in the retained
MVP. Data Market is a real secondary lane, not the new primary onboarding path.

## What Ships In v0.2

The current launch includes:

- a dedicated `Data Seller` pane
- a read-only `Data Market` pane
- a narrow `Data Buyer` pane
- `autopilotctl data-market ...`
- `autopilot_headless_data_market`
- repo-owned Data Market skills
- deterministic packaging helpers
- targeted NIP-90 request/result transport
- live authority objects for asset, grant, delivery, and revocation truth

The current canonical objects are:

- `DataAsset`
- `AccessGrant`
- `PermissionPolicy`
- `DeliveryBundle`
- `RevocationReceipt`

The current transport truth is:

- targeted request kind `5960`
- targeted result kind `6960`
- handler/capability kind `31990`

The strict public-relay verification path now works against:

- `wss://relay.damus.io`
- `wss://relay.primal.net`

## What Is In The UI

The UI now has three distinct Data Market surfaces.

### `Data Seller`

This is the main authoring and fulfillment surface.

It is where the current MVP seller flow lives:

- conversational seller drafting
- structured asset and grant draft state
- exact preview of what will be published
- explicit confirm-before-publish discipline
- incoming request intake and evaluation
- payment-required issuance
- delivery preparation and delivery publication
- revoke and expire controls

This is the primary in-app mutation surface for the Data Market MVP.

It also now doubles as the main visualization surface for shell-first and
headless work, because it shows:

- package metadata
- preview state
- request relay and kind context
- payment state
- delivery truth
- revocation truth

### `Data Market`

This pane is read-only by design.

It is the market snapshot and operator-facing lifecycle surface. It shows:

- published assets
- grants
- deliveries
- revocations
- packaging posture
- visibility and sensitivity posture
- recent seller-side lifecycle activity

This is the observability pane, not the main authoring pane.

### `Data Buyer`

This is a narrow buyer-side MVP surface.

It can:

- refresh the current market snapshot
- select a visible asset or offer
- show bundle and posture context before publication
- publish a targeted request
- track feedback and results
- drive the local consume path for a delivered bundle

This is intentionally not a full buyer procurement workstation.

## What Ships In `autopilotctl` And Headless

`v0.2` is not UI-only.

The same seller and buyer logic is exposed through the app-owned control plane,
which means the current launch also includes a real shell-first path.

### `autopilotctl data-market ...`

The current command tree includes:

- `seller-status`
- `seller-prompt "<plain-language instruction>"`
- `buyer-status`
- `buyer-refresh`
- `snapshot`
- `draft-asset --file ...`
- `preview-asset`
- `publish-asset --confirm`
- `draft-grant --file ...`
- `preview-grant`
- `publish-grant --confirm`
- `request-payment --request-id ...`
- `prepare-delivery --request-id ... --file ...`
- `issue-delivery --request-id ...`
- `revoke-grant --request-id ... --action revoke|expire --confirm`
- `buyer-publish-request`
- `seller-import-request`
- `buyer-import-response`
- `consume-delivery --output-dir ...`

That matters because the CLI is not a second implementation. It drives the
same app-owned state machine as the visible app.

The new `seller-prompt` action is specifically for terminal-driven automation
of the documented conversational seller flow. It sends plain-language seller
instructions into the same dedicated `Data Seller` lane and uses the same
seller skills and typed market tools that the visible pane uses.

### `autopilot_headless_data_market`

For operators, scripts, and agents, `v0.2` also ships a no-window runtime:

- start `autopilot_headless_data_market`
- let it write the normal desktop-control manifest
- point `autopilotctl` at that manifest
- run the same seller and buyer flow without opening the UI

This is the path for:

- automation
- CI-style verification
- scripted publication
- shell-first seller workflows
- agent-driven headless control

The important truth constraint stays the same in headless mode:

- preview before publish
- explicit confirm before publish or revoke
- read authority state back after mutation

## What Ships As Agent Skills

`v0.2` also includes first-party Data Market skills in this repo.

### `autopilot-data-seller`

Use this when an agent is operating inside the visible `Data Seller` pane.

Its job is:

- normalize seller intent into the structured draft
- ask only for missing listing facts
- keep preview and publish discipline honest
- keep delivery and revocation claims tied to real authority state

### `autopilot-data-market-control`

Use this when an agent needs the typed Data Market tool contract.

Its job is:

- keep the agent on `openagents.data_market.*` tools
- treat kernel objects as economic truth
- keep read-back discipline after mutations

### `autopilot-data-seller-cli`

Use this when the job is shell-first or headless.

Its job is:

- package local material for sale
- drive `autopilotctl data-market ...`
- work through the headless runtime when needed
- package redacted Codex conversations safely

If the job is “sell this folder” or “publish this conversation bundle” without
opening the visible UI, this is the skill to point an agent at.

## How To Point Agents At Stuff

This launch includes a real agent story, but it is specific.

Do not point agents at the Data Market by telling them to vaguely click around
the app. Point them at the correct surface and the correct artifact boundary.

### If the agent is working in the UI

Point it at the `Data Seller` pane and tell it to use the seller flow.

What to give it:

- what is being sold
- the local source boundary
- the intended buyer if the grant is targeted
- the price posture
- the policy posture
- any delivery or revocation constraints

In the visible seller lane, the app already auto-provisions the first-party:

- `autopilot-data-seller`
- `autopilot-data-market-control`

That means the correct in-app instruction is essentially:

- use the `Data Seller` pane
- fill the missing seller facts
- preview before publish
- confirm before publish

### If the agent is working shell-first

Point it at the CLI skill and the exact local path boundary.

Best inputs for a shell-first agent are:

- a source path to a file or folder
- a desired title
- a target consumer identity if the listing is private or targeted
- a price in sats if this is not zero-price
- the desired policy template
- a manifest path if a headless runtime is already running

The best instruction shape is:

- use `skills/autopilot-data-seller-cli/`
- package the local material first
- publish through `autopilotctl data-market ...`
- read back seller status and snapshot after each mutation

### If you already have packaged artifacts

Do not make the agent rediscover the package boundary.

Point it directly at:

- `listing-template.json`
- `grant-template.json`
- `packaging-summary.json`

This is the cleanest handoff when packaging has already happened and the agent
only needs to publish or manage lifecycle state.

### If the asset is a Codex conversation bundle

Point the agent at the dedicated redaction and packaging path instead of raw
rollout files.

Use:

- `skills/autopilot-data-seller-cli/scripts/package_codex_conversations.sh`

This is the right path when the source material lives under `~/.codex/sessions`
or when you want a redacted conversation bundle rather than packaging raw
session files directly.

### If the agent is resolving or consuming a delivery

Point it at:

- the `desktop-control` manifest for the target runtime
- the `request_id`, `grant_id`, `asset_id`, or `delivery_bundle_id`
- the buyer output directory

That lets it run `consume-delivery` cleanly without guessing which delivery to
materialize.

### Example agent asks

Good agent instructions for this launch look like this:

- "Use `autopilotctl data-market seller-prompt` against my running manifest and
  tell the `Data Seller` lane to turn `./research/interviews` into a saleable
  listing for `250 sats`, targeted to `npub1...`, then preview before publish."
- "Use `skills/autopilot-data-seller-cli/` to package `./data/customer-ledger`
  as a targeted listing for `npub1...`, price it at `500 sats`, and publish the
  asset and grant through `autopilotctl`."
- "In the `Data Seller` pane, turn `./research/interviews` into a saleable
  listing. Ask me only for the missing price, consumer, and policy facts, then
  preview before publish."
- "Use the redacted Codex conversation packager on recent sessions under
  `~/.codex/sessions`, publish the resulting listing and grant, and read back
  seller status after each step."

## Typical v0.2 Flows

### UI-first seller flow

1. Open the `Data Seller` pane.
2. Describe what you want to sell.
3. Let the seller lane normalize that into the structured draft.
4. Preview the asset.
5. Confirm and publish the asset.
6. Draft, preview, confirm, and publish the grant.
7. Bring the seller online for targeted request intake.
8. Handle payment, delivery, and revocation from the same seller lane.

### Shell-first seller flow

1. Start `autopilot_headless_data_market` if you do not want the UI window.
2. Either package the local material with the deterministic packager or send a
   plain-language instruction through `autopilotctl data-market seller-prompt`.
3. If you packaged first, use `autopilotctl data-market draft-asset --file ...`.
4. Run `preview-asset`.
5. Run `publish-asset --confirm`.
6. Repeat the same draft, preview, publish flow for the grant.
7. Continue with request handling, payment, delivery, revoke, or consume.

### Narrow buyer flow

1. Refresh the market.
2. Select a visible asset or offer.
3. Publish a targeted request.
4. Track feedback and results.
5. Consume the delivered payload locally.

## Important Scope Boundaries

This launch is real, but it is deliberately narrow.

`v0.2` is not:

- a broad searchable data marketplace
- a rich public catalog and comparison product
- a general buyer procurement workstation
- a remote blob retrieval client
- a replacement for the compute-first onboarding promise

Current important limitations:

- requests are targeted by default
- the `Data Market` pane is read-only
- the buyer product is still narrow
- local consume currently supports local `file://` refs and plain local paths
- the Data Market is secondary to the compute-provider loop in overall product
  positioning

## Operator And Trust Notes

The important architecture rule for `v0.2` is:

- UI, CLI, headless, and skills all drive the same app-owned Data Market logic

They do not publish through parallel implementations.

Economic truth lives in kernel-backed objects:

- `DataAsset`
- `AccessGrant`
- `DeliveryBundle`
- `RevocationReceipt`

Transport truth for targeted request and result traffic lives on the NIP-90
data-vending profile.

That split is what makes the launch honest:

- the relay carries requests and results
- the kernel carries the economic record
- the app-owned state machine makes the flow usable across UI, CLI, and agents

## Bottom Line

`v0.2` is the first real OpenAgents Data Market release.

It lets sellers package local material, publish truthful asset and grant state,
accept targeted requests, deliver a real bundle, and revoke access when needed.

It also ships in a way that is actually usable:

- visible UI for interactive use
- `autopilotctl` for shell-first control
- a headless runtime for operators and scripts
- first-party skills for in-app and shell-first agents
- a proven one-sentence `Data Seller` prompt -> paid buy -> delivery -> consume
  audit under `docs/audits/2026-03-21-data-seller-one-sentence-prompt-paid-flow-audit.md`

That is the right scope for this launch.

It is more than a spec, but less than the final marketplace vision.
